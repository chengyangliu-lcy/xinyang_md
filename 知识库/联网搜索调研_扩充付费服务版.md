# 联网搜索调研

## 背景与目标

当前 Agent 需要具备两类联网能力：一类是 **Web Search** ，用于根据问题检索候选网页；另一类是 **Web Fetch / Web Read** ，用于读取候选网页正文并交给 LLM 分析。现有实现主要依赖直接爬取搜索引擎 HTML 页面，以及用 `requests + BeautifulSoup` 读取网页正文。这套方案成本低，但稳定性较差，遇到搜索引擎反爬、JS 渲染页面、Cloudflare 校验页、403/429 限流等情况时容易失败。

### 2.2 推荐：SearXNG 自部署

![pic](https://cdn.nlark.com/yuque/0/2026/png/43223830/1778642070416-ddafa55e-a5d4-4c44-acf4-bd0d24ffb190.png)

**为什么是最佳选择：**

  * 完全免费开源（AGPL），Docker 一行命令部署


  * 聚合 70+ 搜索引擎（Google、Bing、DuckDuckGo、Wikipedia 等）


  * 天然分散反爬风险 — 不依赖单一引擎


  * 提供 JSON API，直接返回结构化结果


  * 无调用限制，无 API Key


  * LangChain、LlamaIndex 等框架有内置支持


**部署方式：**
```text
    docker run -d --name searxng -p 8888:8080 \
      -e SEARXNG_BASE_URL=http://localhost:8888 \
      searxng/searxng
```

**调用方式：**
```text
    import requests
    
    def search_searxng(query: str, limit: int = 10) -> list[dict]:
        resp = requests.get(
            "http://localhost:8888/search",
            params={"q": query, "format": "json", "categories": "general"},
            timeout=15
        )
        results = resp.json().get("results", [])
        return [
            {"title": r["title"], "url": r["url"], "snippet": r.get("content", "")}
            for r in results[:limit]
        ]
```

**注意事项：**

  * 需要一台服务器（VPS 或本地 Docker）


  * 上游搜索引擎仍可能限制高频请求，建议加缓存


  * 可通过配置文件调整引擎权重和超时

**免费额度最高的官方 API：**

  * 1,000 次/月，每月重置


  * 注册即得 API Key：[https://api.search.brave.com](https://api.search.brave.com)


  * 返回结构化 JSON，支持 web/images/news 搜索


```text
    import requests
    
    def search_brave(query: str, api_key: str, limit: int = 10) -> list[dict]:
        resp = requests.get(
            "https://api.search.brave.com/res/v1/web/search",
            headers={"X-Subscription-Token": api_key},
            params={"q": query, "count": limit},
            timeout=15
        )
        results = resp.json().get("web", {}).get("results", [])
        return [
            {"title": r["title"], "url": r["url"], "snippet": r.get("description", "")}
            for r in results
        ]
```

### 3.3 推荐：Jina Reader 作为远程兜底层

**最简单的网页抓取方案：**

只需在目标 URL 前加 `https://r.jina.ai/`，即可获取干净的 Markdown 内容：
```text
    import requests
    
    def fetch_jina(url: str, max_chars: int = 8000) -> str:
        resp = requests.get(
            f"https://r.jina.ai/{url}",
            headers={"Accept": "text/plain"},
            timeout=20
        )
        return resp.text[:max_chars]
```

**特点：**

  * 无需 API Key（基础版免费）


  * 服务端处理 JS 渲染和反爬


  * 返回干净的 Markdown，直接给 LLM 消费


  * 支持 header 控制：`X-Return-Format`、`X-Wait-For-Selector` 等


  * 也有搜索服务：`https://s.jina.ai/{query}`


**限制：**

  * 免费版有速率限制


  * 不适合大批量抓取

###  4.3 搜索服务价格梯度：谁便宜，谁适合生产

从单价看，如果只比较“候选网页搜索”这一层，成本大致可以分成四档：

成本档位| 服务| 大致价格| 适合场景  
---|---|---|---  
**极低成本**|  SearXNG 自部署、DuckDuckGo 库、Jina Reader Search、DashScope 中国内地/Global Web Search| $$0 或低于 $$1 / 1k 级别（DashScope 中国内地/Global 为 $0.57341 / 1k invocations）| 内部实验、周会 demo、低成本 Agent 原型  
**中低成本**|  Brave Search API、Google Custom Search 既有客户、OpenAI / Exa Answer| $$5 - $$10 / 1k| 通用搜索、问答引用、一般生产  
**中高成本**|  Tavily PAYG、Exa Search、Bing Grounding| $$7 - $$14 / 1k；Tavily PAYG 为 $8 / 1k credits| 需要 LLM 友好结果、语义搜索、Azure 集成  
**高成本 / 专项成本**|  SerpApi、Browserbase、Browserless、Firecrawl 复杂提取| SerpApi 约 $$9 - $$25 / 1k search；浏览器服务按 unit/hour 计费| Google SERP、Scholar、Maps、登录态、JS 交互、复杂网页  
  
这里要注意一个关键点：**Web Search 和 Web Fetch 的成本结构完全不同** 。搜索 API 通常按“query/request”计费；网页读取则按“page、credit、unit、browser hour、proxy traffic”计费。Agent 如果每个问题搜索 3 次、每次读 5 个网页，那么一次用户请求可能变成：
```text
    1 次用户问题
      = 3 次 search API 调用
      + 15 次 web fetch / page scrape
      + 15 个页面正文 token 输入 LLM
```

所以真实成本往往不是“搜索一次多少钱”，而是：
```text
    总成本 = 搜索成本 + 页面抓取成本 + 浏览器渲染成本 + 代理成本 + LLM 输入/输出 token 成本
```

### 4.4 典型月用量成本估算

下面按常见 Agent 使用规模粗算，帮助判断是否值得接入付费 API。

#### 场景 A：周会 demo / 内部实验

假设每月：

  * 500 - 1,000 次搜索


  * 每次搜索读取 1 - 3 个网页


  * 不要求强 SLA


推荐组合：
```text
    SearXNG 自部署 / DuckDuckGo 库
      + requests/BeautifulSoup
      + Crawl4AI 本地
      + Jina Reader 兜底
      + Brave 或 Tavily 免费额度兜底
```

预计成本：

项目| 成本  
---|---  
SearXNG / DuckDuckGo| $0  
Crawl4AI 自部署| 机器已有则 $0，独立 VPS 另算  
Jina Reader| $0 起  
Brave/Tavily 免费额度| $0  
**合计**| **基本 $0**  
  
适合当前阶段做原型验证，不建议一开始就上重付费方案。

#### 场景 B：小规模生产 Agent

假设每月：

  * 10,000 次搜索


  * 20,000 次网页读取


  * 要求比较稳定，但不涉及登录态和复杂浏览器交互


可选组合 1：Brave + Firecrawl

项目| 估算  
---|---  
Brave Search：10k requests| 约 $50  
Firecrawl Standard：100k credits/月| $83/月（年付口径）  
**合计**| **约 $133/月**  
  
可选组合 2：Tavily + Firecrawl

项目| 估算  
---|---  
Tavily PAYG：10k credits| 约 $80  
Firecrawl Standard：100k credits/月| $83/月（年付口径）  
**合计**| **约 $163/月**  
  
可选组合 3：Exa Search + Exa Contents

项目| 估算  
---|---  
Exa Search：10k requests| 约 $70  
Exa Contents：20k pages| 约 $20  
**合计**| **约 $90/月**  
  
这个场景下，**Brave 更便宜稳妥，Tavily 更偏 Agent 友好，Exa 更偏语义检索和代码/文档场景** 。

#### 场景 C：复杂网页、登录态、动态 JS、网页操作型 Agent

假设每月：

  * 需要打开网页、点击、等待 JS 渲染


  * 可能要登录、保持 session、截图或解决部分 bot 检测


  * 每个任务可能运行几十秒到几分钟


推荐组合：
```text
    Brave / Tavily / Exa 负责搜索候选 URL
      + Browserbase 或 Browserless 负责浏览器执行
      + Firecrawl / Crawl4AI 负责普通页面抽取
```

成本判断：

服务| 起步成本| 适合  
---|---|---  
Browserbase Developer| $20/月，含 100 browser hours| 轻量浏览器 Agent、Stagehand、登录态测试  
Browserbase Startup| $99/月，含 500 browser hours| 小规模生产 Agent  
Browserless Prototyping| $25/月，含 20k units| Playwright/Puppeteer 服务化  
Browserless Starter| $140/月，含 180k units| 稳定浏览器自动化服务  
  
这种场景的核心不是搜索质量，而是 **浏览器会话稳定性、并发、代理、cookie/session 管理、captcha 处理和失败重试** 。如果只是读普通新闻、博客、文档，直接用 Browserbase/Browserless 会过重。

#### 场景 D：模型原生联网搜索

如果你希望模型自己决定何时搜索、如何综合多个来源，而不是你手写 `search -> fetch -> summarize` 流程，可以考虑模型原生联网工具。

方案| 价格口径| 适合| 不适合  
---|---|---|---  
OpenAI Web Search Tool| $$10 / 1k calls + search content tokens 按模型费率；部分 preview 旧工具 $$25 / 1k calls| 高质量问答、需要引用、需要模型主动搜索| 只想拿 URL 列表、低成本批量抓取  
DashScope / Qwen Web Search| 中国内地和 Global $$0.57341 / 1k invocations；International $$10 / 1k invocations；另加模型 token| 已经使用 Qwen/DashScope，想低成本接入搜索| 需要跨模型通用、需要完全控制搜索结果排序  
Azure Grounding with Bing| $14 / 1k transactions| Azure AI Foundry Agent 内部使用| 想替代旧 Bing Search API、想直接拿原始 SERP  
  
### 4.5 推荐落地架构：分层 fallback，而不是只选一个服务

建议把联网能力拆成两层：
```text
    Search 层：负责找候选网页
    Fetch 层：负责读取候选网页正文
```

推荐实现如下：
```text
    web_search(query)
      1. SearXNG 自部署                 # 免费主路径，适合内部和低频
      2. Brave Search API               # 官方稳定兜底，成本可控
      3. Tavily / Exa                   # Agent 友好或语义搜索增强
      4. SerpApi                        # 只有确实需要 Google/Scholar/Maps SERP 时启用
      5. OpenAI / DashScope 原生搜索      # 问答型场景，不作为通用 URL 搜索接口
    
    web_fetch(url)
      1. requests + BeautifulSoup        # 最轻量，优先尝试
      2. Crawl4AI 本地 Playwright         # JS 渲染主路径
      3. Jina Reader                     # 远程 Markdown 兜底
      4. Firecrawl                       # 生产级抓取、整站 crawl、结构化抽取
      5. Browserbase / Browserless       # 登录态、点击、动态交互、截图、复杂网页
```

这样设计的好处是：

  * **成本可控** ：普通问题走免费/低价路径，疑难页面才用付费浏览器。


  * **稳定性更强** ：某个搜索源被限流，不会导致整个 Agent 失效。


  * **可观测性更好** ：可以记录每层成功率、耗时、成本，再决定是否升级。


  * **可解释性更强** ：Search 返回候选 URL，Fetch 返回正文，LLM 只做分析与综合。


### 4.6 推荐配置表

使用阶段| Search 推荐| Fetch 推荐| 月成本预期| 说明  
---|---|---|---|---

### 4.7 预算控制建议

为了避免 Agent 联网成本失控，建议在工程上加以下限制：

  1. **搜索缓存** ：对 query 做 normalization，例如小写、去空格、去时间戳，把搜索结果缓存 1 - 24 小时。


  2. **URL 去重** ：同一轮任务中相同 URL 只 fetch 一次。


  3. **限制读取页数** ：默认只读取 Top 3；只有答案置信度不足时再扩展到 Top 5 / Top 10。


  4. **限制正文长度** ：每个网页最多截取 8k - 20k chars，避免把整站内容塞进 LLM。


  5. **区分 cheap fetch 和 expensive fetch** ：普通页面先 `requests`，失败后 Crawl4AI，再失败才 Firecrawl/Browserless。


  6. **按任务设置预算** ：例如单个用户问题最多 3 次 search、最多 5 次 fetch、最多 1 次 browser session。


  7. **记录成本日志** ：每次工具调用记录 provider、latency、success、tokens、estimated_cost。


  8. **配置熔断** ：某个 API 429/403/5xx 超过阈值后自动切换 provider。


  9. **设置 billing alert** ：Brave、Tavily、Exa、OpenAI、Azure、Browserbase 都需要设置月度预算提醒。


可以在代码里加一个简单的预算对象：
```text
    from dataclasses import dataclass
    
    @dataclass
    class WebBudget:
        max_search_calls: int = 3
        max_fetch_pages: int = 5
        max_browser_sessions: int = 1
        max_total_chars: int = 50000
    
    
    def should_continue_search(used_search_calls: int, budget: WebBudget) -> bool:
        return used_search_calls < budget.max_search_calls
    
    
    def should_fetch_more(fetched_pages: int, total_chars: int, budget: WebBudget) -> bool:
        return fetched_pages < budget.max_fetch_pages and total_chars < budget.max_total_chars
```

### 4.8 最终推荐方案

如果目标是给当前 Agent 增加稳定联网能力，推荐不要直接购买很多服务，而是按阶段推进：

**第一阶段：免费优先，补齐链路**
```text
    SearXNG 自部署
      + DuckDuckGo 库 fallback
      + requests/BeautifulSoup
      + Crawl4AI
      + Jina Reader fallback
```

这个阶段主要验证：Planner 是否会正确选择 `web_search` / `web_read`，搜索结果是否能进入后续分析，网页正文是否能被正常清洗成 Markdown。

**第二阶段：接入一个便宜稳定的官方 Search API**

优先接 **Brave Search API** 。原因是：价格清晰，$$5 / 1k requests；每月还有 $$5 免费额度；返回 JSON；不需要自己处理搜索引擎反爬。它最适合作为 SearXNG 失败后的稳定兜底。

**第三阶段：按需求接入专项能力**

  * 如果 Agent 问答效果不够好：接 Tavily。


  * 如果代码文档、公司信息、语义搜索多：接 Exa。


  * 如果网页读取失败率高：接 Firecrawl。


  * 如果需要登录态、点击、截图、复杂 JS：接 Browserbase 或 Browserless。


  * 如果完全在 Qwen/DashScope 生态内：优先评估 DashScope Web Search / Web Extractor。


  * 如果需要模型自主搜索和带引用回答：评估 OpenAI Web Search Tool 或 Azure Grounding。


**不推荐当前直接作为主路径的方案：**

  * **Google Custom Search JSON API** ：新客户不可用，并且官方已给出停止时间。


  * **旧 Bing Search API** ：已经退役，只能考虑 Azure Grounding，但它不是通用 SERP API。


  * **SerpApi 作为默认主路径** ：能力强但价格偏高，更适合作为 Google/Scholar/Maps 等特殊场景兜底。
