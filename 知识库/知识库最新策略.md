# ComponentAnomalyAgent 知识库检索策略优化

  


[https://www.electronicsforu.com/](https://www.electronicsforu.com/)

## 一、背景与目标

项目是一个面向电子电路异常诊断的 RAG 问答系统。用户输入中文技术问题（可附带电路板图片），系统检索本地知识库和网络证据，生成结构化中文答案，并与参考答案进行多维度自动评分。目标是在 baseline（final_score=0.5789）基础上通过知识库策略提升答案质量。

## 二、旧系统问题诊断

旧 KB 存在以下问题：

  * **规模过大、噪声严重** ：57,506 chunks，来自 elecfans 等论坛爬取，大量低价值内容、水文和模板文本。
  * **检索质量差** ：仅做简单 Dense+Sparse 混合，无重排序，低相关性 chunk 混入证据。
  * **证据利用率极低** ：KB 检索结果不被写入最终答案（planner prompt 明确指示 "KB 证据不会进入最终答案"）。
  * **答案模板缺失** ：自由格式输出导致答案松散、采分点命中率低。
  * **幻觉严重** ：LLM 编造题面中不存在的元器件标号和参数值。


## 三、主要工作

### 3.1 知识库重建（v2 KB）

  * 对 57k 旧 KB 进行严格清洗过滤。
  * **去除** ：低价值来源（非技术 URL）、水文/模板文本、低价值项目、与问题无关的 chunk。
  * **最终结果** ：保留 5,924 chunks / 5,584 docs，规模缩小 90%。
  * **新 KB 路径** ：`knowledge_base/circuit_diagnosis_fts_hq_v2`


### 3.2 检索 Pipeline 升级

从简单的 Dense+Sparse 混合检索升级为 6 阶段 Pipeline：

**阶段**| **方法**| **配置**  
---|---|---  
**Query Rewriting**|  LLM 将中文题面改写为英文电子关键词| 保留型号/位号/数值  
**Dense 检索**|  Qwen3-Embedding-4B (2560-dim)| 权重 65%  
**Sparse 检索**|  SQLite FTS5 + BM25| 权重 35%  
**Hybrid 融合**|  Dense × 0.65 + Sparse × 0.35| threshold=0.15  
**Cross-encoder 重排**|  BAAI/bge-reranker-v2-m3| top-30 候选精排  
**Contextual Retrieval**|  chunk 前预置文档上下文| 提升匹配精度  
  
**涉及文件** ：`tools/circuit_kb.py`、`tools/dense_retriever.py`

### 3.3 答案生成优化

  * **恢复结构化模板（1-5 编号格式）** ：


1.结论 → 2.依据 → 3.原因机制 → 4.检查步骤 → 5.处理建议与不确定性

  * **强化反幻觉机制** ：


  * 强制因果链输出（如 "R161+C24 串联→容抗增大→高频增益下降"），禁止 "起到滤波作用" 等泛化表述。
  * 不得编造题面和证据中均未出现的元件标号、参数值或型号。
  * 图片与 KB 矛盾时采信图片并标注冲突。


  * **KB 证据入答案** ：planner prompt 从 "KB 证据不进入答案" 改为 "KB 证据经 rank_evidence 排序后进入最终答案"。
  * **答案清理** ：`_sanitize_answer` 真正删除编造内容（匹配并移除不在题面/证据中的元件标号和相关参数），而非仅在末尾标注。


**涉及文件** ：`agent/prompts.py`、`tools/evidence_tools.py`

### 3.4 API Provider 系统适配

  * 新增 provider 抽象层，支持 dashscope / vllm / openai。
  * 每个 provider 独立配置 `base_url`、`api_key`、`extra_body`。
  * 从本地 vllm (claude-sonnet-4-6) 切换到 DashScope API (qwen3.6-plus)。
  * 配置通过 `configs/local.yaml` 管理，不进入 git。


**涉及文件** ：`configs/config.py`、`configs/default.yaml`、`configs/local.yaml`

### 3.5 实验设计与消融分析

设计 4 组对照实验，逐模块消融贡献：

**实验**| **配置**| **目的**  
---|---|---  
**Exp12**|  无 KB + 灵活模板| 纯模型 baseline  
**Exp11**|  v2 KB only + 灵活模板| 孤立 KB 贡献  
**Exp10**|  v2 KB + 全检索特性 + 灵活模板| 检索特性贡献  
**Exp13**|  v2 KB + 全检索特性 + 结构化模板| 模板贡献  
  
全量 368 样本，5 并发推理+评估，每组实验跑通。实验配置统一管理在 `configs/experiments/` 目录。

### 3.6 修复与工程改进

  * **修复 LLM Judge JSON 截断** ：三层兜底（直接解析→JSON修复→正则提取），`judge_max_tokens` 从 2000 提升到 4000。
  * **修复并发问题** ：修复 `append_jsonl` 并发写入重复行问题，添加去重脚本。
  * **指标修正** ：修复 `kb_evidence_used_rate` 指标（虽然后续发现检测方式仍有缺陷）。
  * **稳定性提升** ：修复 DashScope 批量推理 API 扣费失败导致的 42 条全部 fallback 问题。
  * **新增辅助工具** ：`scripts/dedupe_jsonl.py`、`scripts/repair_eval.py`、`scripts/compare_runs.py`。

## 五、结论与展望

用 1/10 规模的 v2 KB + 检索 Pipeline + 结构化模板，在全部 368 条样本上追平并微弱反超旧系统。核心发现：

  * **KB 内容质量 > 数量**：清洗后的 5,924 chunks 比原来的 57k chunks 更有效。
  * **答案模板 > 检索复杂度**：结构化模板贡献了单模块最大的分数提升（+0.007），超过所有检索特性的总和。
  * **检索特性独立生效必须依赖 KB 质量** ：KB alone 反而有害，必须配合 cross-encoder 和 query rewriting 才能发挥价值。
  * **后续方向** ：KB 内容应从通用电路文章转向针对性更强的 datasheet/app note；证据到答案的转换效率（LLM 如何把 chunk 变成因果链）是最大瓶颈。
