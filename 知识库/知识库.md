# 知识库

## 当前知识库策略与实验对比总结

### 知识库策略架构

**数据源** ：电路诊断技术文档（从 elecfans.com 等电子技术网站爬取的 Markdown），存储在 SQLite FTS 索引中（12,111 条候选 chunks）。

**检索方式** ：混合检索（Dense 65% + Sparse 35%）

  * Dense：Qwen3-Embedding-4B 向量检索，语义匹配
  * Sparse：SQLite FTS5 + BM25 重排序，关键词匹配
  * 合并后按 hybrid score 排序，每 URL 最多取 2 chunks


**多层过滤** ：

  1. 低价值来源过滤（非技术类 URL）
  2. 水文/模板文本过滤
  3. 低价值项目过滤（与 query 不匹配的项目文档）
  4. Required terms 过滤（query 关键词必须在结果中出现）
  5. 低相关性过滤（sparse score < 5.0 或 hybrid score < 0.15）


**调用门槛** （_should_use_local_retrieval）：

  * 必须同时满足：有强实体（型号/位号/参数值）+ 有故障/公式/拓扑需求
  * 如果有图片证据但缺上述条件，优先用图片而跳过 KB
  * 每样本最多调用 1 次，返回最多 4 chunks


### 实验对比

### KB 使用诊断

指标| 值  
---|---  
KB 检索触发率| 167/368 = **45.4%**  
检索非空率| 49.1%（约一半触发后查不到相关内容）  
平均返回 chunks| 1.7  
KB 证据被用于最终答案| **0.27%** （仅 1 个样本）  
候选 → 高相关性| 283/2329 = 12.2%（大量被过滤）  
  
### 核心结论

知识库几乎**没有带来提升** （final_score 差 0.004），原因有三：

  1. **检索门槛太严** ：只在"强实体 + 故障/拓扑"条件下触发，45% 的题目根本不调 KB
  2. **过滤太狠** ：候选 2329 条，过滤后仅保留 283 条（88% 被丢弃），检索非空率只有 49%
  3. **知识利用率为 0** ：agent 几乎不把 KB 内容写入最终答案（kb_evidence_used_rate = 0.27%），说明 planner 拿到 KB 内容后没有有效融入回答


最可能的改进方向：放宽检索触发条件和过滤策略，让更多样本能拿到 KB 内容；同时优化 planner prompt，让它更积极地将检索到的技术知识融入答案

## 二、预处理策略

### 2.1 Markdown 解析 (`parse_markdown_page`)

  * 从 `**标题**`/`**链接**`/`**发布时间**` 标签提取元数据
  * 文本 body 做 SHA1 哈希用于去重
  * 标题和内容做 `compact_text` 截断


### 2.2 多层过滤管道

#### Phase 0: 文件级过滤

  * 超大文件（>2MB）直接跳过
  * 只处理 `page_N.md` 格式文件


#### Phase 1: 元数据快速过滤 (`build_diagnosis_kb`)

  * **低价值来源过滤** (`is_low_value_source`): 过滤 `circuitmaker.com/user/`、`circuitmaker.com/forum/` 等
  * **标题过滤** : 过滤 "404 页面不存在"、"Page not found"、"提示信息" 等垃圾标题
  * **URL 路径过滤** : 对 elecfans 等中文源，要求 URL 在允许路径中（`/article/`、`/dianlutu/`、`/analog/` 等），拒绝 `/tag/`、`/category/`、`/news/` 等
  * **快速诊断标记检查** : 对非白名单源，检查 title+URL 是否包含 `CIRCUIT_DIAGNOSIS_CONTENT_MARKERS`（如"故障分析"、"datasheet"、"TL431" 等 67 个标记）


#### Phase 2: 深度内容过滤

  * **中文源** (`elecfans`/`eet-china`):


  1. `is_public_tech_kb_page`: 检查技术术语密度（>=2 个电子术语或 >=1 个高价值标记）
  2. `_clean_public_tech_page`: 去除导航、推荐、下载等 boilerplate 段落
  3. `is_deep_circuit_page`: 要求 2+ 个诊断标记，或有型号+电路上下文
  4. `is_useful_page`: 字符比例检查（字母+中文 >= 35%）


  * **非中文源** (radiokot, hackster 等):


  1. `is_useful_page`: 基础质量检查
  2. `is_deep_circuit_page`: 深度电路内容检查


#### Boilerplate 清洗 (`clean_public_tech_text`)

  * 识别"文章标题"heading 作为正文起点
  * 遇到"相关技术文章"、"相关资料下载"、"热门推荐" 等 20 个停止标记时截断
  * 逐行过滤: 导航行、版权信息、登录注册、短行(<8字符无数字)、图片引用
  * 去重: URL 相同或 text_hash 相同的文档去重


### 2.3 Chunking 策略 (`chunk_text`)

参数| 默认值  
---|---  
`chunk_chars`| 900 字符  
`chunk_overlap`| 120 字符  
`min_chunk_chars`| 160 字符  
  
  * 先对文本做规范化（去除空行 + strip）
  * 短文本（<160字符）丢弃
  * 中等文本（<=900字符）整体作为一个 chunk
  * 长文本在句子边界处切分（`\n`、`. `、`。`、`；`），确保 chunk > 450 字符（一半 `chunk_chars`）时才在边界切


### 2.4 High-Quality KB 构建 (`build_hq_kb.py`)

更严格的构建管线，使用多维质量评分：

**质量评分维度** (总分 >= 3 才保留):

  1. **诊断标记** (最高 5 分): 从 `CIRCUIT_DIAGNOSIS_CONTENT_MARKERS` 命中数
  2. **技术深度** (最高 5 分): 公式、波形、Bode 图等 `TECHNICAL_DEPTH_MARKERS` 命中
  3. **元件型号** (最高 3 分): TL431、LM358 等 60+ 个常见型号
  4. **原理图引用** (最高 3 分): "原理图"、"schematic"、元件位号等
  5. **内容结构** (最高 3 分): 长度 > 2000、有 ## 标题、编号列表
  6. **因果推理** (最高 3 分): "因为"、"导致"、"causes" 等因果标记
  7. **QA 结构** (最高 3 分): "请问"、"如何解决"、"how to" 等
  8. **惩罚项** : 低价值源 -5、boilerplate -3、过短 -3、导航残留 -2 等
  9. **特定域调整** : elecfans 标题-内容不匹配 -3、radiokot 非电路路径 -5 等


**额外特性** :

  * 多文章页面拆分: 对 elecfans 合并页面按"发表于 日期"分割
  * 子文章选择: 选择标题关键词重叠度最高的子文章
  * Boilerplate 深度清洗: 按域名使用数十个正则清洗导航/下载/论坛交互
  * Chunk 后质量过滤: < 500 字符的 chunk 必须包含电路关键词才保留
  * 上下文前缀: 每个 chunk 前添加 `[文档: 标题] [来源: 域名]` 用于增强检索匹配

## 三、检索策略

### 3.1 检索引擎: `CircuitMarkdownRetriever`

#### 架构: 混合检索 (Hybrid Search)

组件| 实现| 配置  
---|---|---  
**稀疏检索**|  SQLite FTS5 + trigram tokenizer + BM25| `candidate_limit=80`  
**稠密检索**|  Qwen3-Embedding-4B (2560维) + 余弦相似度| `dense_weight=0.65`  
**交叉编码器**|  BAAI/bge-reranker-v2-m3 (可选)| `cross_encoder_top_n=30`  
  
#### 检索流程:

  1. **FTS 查询构建** (`build_fts_query`):


  * 从 query 中提取术语并按优先级排序: models > refdes > values > topology > fault > other
  * 如果有强术语（型号/位号/数值），用 OR 连接前 6 个
  * 否则用 OR 连接前 12 个术语
  * 过滤宽泛术语 (`BROAD_QUERY_TERMS`): "ac", "dc", "circuit", "power" 等 35 个


  2. **术语提取** (`extract_query_terms`):


  * 正则匹配: 字母/数字/CJK + 特殊符号
  * Jieba 分词补充 (`retriever.tokenize`)
  * **中英文术语扩展** (`expand_chinese_electronics_terms`): 50+ 个中英对照对，如 "电流采样" → "current sense", "sense resistor", "shunt"


  3. **术语分类** (`classify_query_terms`):


  * `models`: 型号规则匹配 (如 `TL431`)
  * `refdes`: 位号匹配 (如 `R5`, `C12`)
  * `values`: 数值+单位 (如 `10kΩ`, `5V`)
  * `topology`: 拓扑词 (buck, boost, flyback 等 30 个)
  * `fault`: 故障词 (burn, overheat, 烧毁, 短路 等 30 个)
  * `other`: 其他技术术语


  4. **稀疏搜索** : FTS5 BM25 查询，失败时回退到 LIKE 查询
  5. **稠密搜索** : Qwen3-Embedding-4B 编码 query → 与 57,506 个 embedding 做点积 → Top-K (K=80)
  6. **混合评分合并** :


```text
    稀疏分数归一化: sparse_norm = score / max(sparse_scores)
    稠密分数归一化: dense_norm = cos_sim (clip 0-1)
    混合分数: hybrid = 0.65 * dense_norm + 0.35 * sparse_norm
```

稠密-only hits (不在稀疏结果中): 要求 `dense_norm >= 0.3` 才补充

  7. **重排序** (`rerank_rows`): 对 BM25 结果做多维评分：


  * 基础分: `1 / (1 + abs(bm25_score))`
  * Boilerplate 惩罚: -8.0
  * 低价值源惩罚: -8.0
  * CircuitMaker 项目惩罚: -2.5 到 -8.5
  * 术语匹配加分: title 中 +3.0、正文 +1.0
  * 型号匹配: +5.0（命中）/-1.5（未命中）
  * 位号匹配: +2.0
  * 拓扑/故障词匹配: +1.0~+2.0
  * 高价值文本标记: +0.8/个
  * 电子术语匹配: +0.6/个
  * 噪声标记惩罚: -3.0
  * 短文本惩罚: -2.0


  8. **交叉编码器重排序** (可选): 用 `bge-reranker-v2-m3` 对候选做 full-text 对比，按 CE 分数重排


### 3.2 多阶段过滤

候选结果经过以下过滤：

  1. **低价值源过滤** : `is_low_value_source` (circuitmaker user/forum 页)
  2. **Boilerplate 过滤** : `is_boilerplate_text` (导航/版权/噪声标记)
  3. **低价值项目过滤** : `is_low_value_project_row` (CircuitMaker 空洞项目)
  4. **必选术语检查** : 如果有强术语（型号/位号/数值），要求 chunk 中必须包含至少一个；否则要求 2+ 高价值匹配
  5. **URL 去重** : 每个 URL 最多保留 N 个 chunks (`max_chunks_per_url=2`，CE模式=3)
  6. **相关性阈值** :


  * 纯稀疏模式: `score >= 5.0`
  * 混合模式: `hybrid >= 0.15`
  * 高相关性标记: hybrid >= 0.4 或 CE >= 0.5


### 3.3 Evidence 输出

最终返回的 chunk 被包装为 `Evidence(kind="local_kb_chunk")`，包含:

  * title, url, content (截断 4000 字符)
  * 元数据: chunk_id, page_id, bm25_score, rerank_score, kb_relevance, high_relevance, matched_query_terms

## 四、Agent 集成策略

### 4.1 触发条件 (`_should_use_local_retrieval`)

**评分制触发** （>= 2.0 分才允许使用 KB）:

信号类型| 计分规则  
---|---  
型号 (models)| `min(3.0, count × 1.5)`  
位号 (refdes)| `min(3.0, count × 1.5)`  
数值 (values)| `min(3.0, count × 1.0)`  
拓扑词 (topology)| `min(2.0, count × 1.0)`  
故障词 (fault)| `min(2.0, count × 1.0)`  
公式/计算/选型关键词| +2.0  
无 web 证据时的替代激励| +1.0  
  
**图片延迟逻辑** :

  * 有未检查图片 + 实体得分 < 4.0 → 不触发 KB（先看图）
  * 有已检查图片 + 实体得分 < 2.0 → 不触发 KB


**调用限制** : 最多 2 次 (`max_local_retrieval_calls=2`)

### 4.2 Planner 引导 (`prompts.py`)

Planner 收到指令:

"local_retrieve 收集本地电路知识库证据，KB 证据会被 rank_evidence 排序并进入最终答案；当题面包含明确型号、位号、数值、拓扑词或故障症状时优先使用。由于本地库主要是英文，调用时要改写成英文电子关键词，并保留型号、位号和数值。"

### 4.3 Evidence 排序 (`EvidenceRankExecutor`)

KB 证据在排序中的加分策略:

  * `domain_skill`: +4.0
  * `image_inspection`: +3.0
  * `local_kb_chunk`: **+1.5** (之前版本直接丢弃 KB 证据，现在参与排序)
  * `_trusted_source`: +2.0（elecfans, eeworld, eet-china 等 10 个域名）


### 4.4 最终答案格式化 (`_format_evidence_for_answer`)

证据按优先级排序后格式化:

类型| 优先级| 字符限制  
---|---|---  
image_inspection| 0| 2200  
domain_skill| 1| 650  
local_kb_chunk| 2| 800  
web_page| 3| 600  
web_search_result| 4| 600  
image_context| 5| 180  
  
KB 证据格式化包含 **谨慎提示** :

"KB 证据来自预建技术文档索引，内容可能存在偏差，使用前需对照题面和图片验证一致性。"

Web 证据的 boilerplate 会被 `_compress_boilerplate` 压缩（优先保留因果句和技术句），KB 证据同样走压缩。

### 4.5 答案生成 (`FINAL_ANSWER_SYSTEM_PROMPT`)

关键规则 9 和 10 指导 KB 证据使用:

  * "KB 证据来自预建技术文档索引，可能文不对题或部分过时；如与图片识别结果矛盾，采信图片并注明冲突"
  * "将 KB 内容自然地融入分析与结论，不必写明'根据本地知识库'"


最终答案生成后会走 `_sanitize_answer` 将题面和证据中不存在的元件标号/型号删除。
