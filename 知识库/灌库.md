/data/xinyang_milvus/volumes/  
├── milvus/ (796M) ← 索引元数据、WAL

步骤：

# 1\. 停止 Milvus

cd /home/music/lcy/material-rag/05_milvus_deploy  
docker compose down

# 2\. 打包

tar czf /data/milvus_backup_$(date +%Y%m%d).tar.gz -C /data/xinyang_milvus volumes/

# 3\. 传输到目标机器

scp /data/milvus_backup_20260522.tar.gz target:/data/

# 4\. 目标机器解压

mkdir -p /data/xinyang_milvus  
tar xzf milvus_backup_20260522.tar.gz -C /data/xinyang_milvus

# 5\. 用相同的 docker-compose.yml 启动

docker compose up -d

优点：零风险，数据完全一致。  
缺点：需要停服，且目标机器的 Milvus 版本要一致（当前 v2.5.14）。

增量灌库标准化流程方案

新 PDF 文档在其他机器上，通过 MinerU API 模式 (mineru -p xxx.pdf -o temp --api-url http://...) 转换为 Markdown + content_list.json 后，需要传到当前机器灌入 Milvus。

当前 V2 灌库脚本 (import_content_list_v2.py) 强依赖 mapping 文件 + Excel 元数据 + tmp_name 目录结构，不适合"外部 PDF 增量进入"的场景。需要一套标准化流程。

整体流程设计

[其他机器] [这台机器]

PDF 文件 → MinerU API → content_list.json  
↓  
scp/rsync 传输  
↓  
MinerU 输出目录 + metadata.csv  
↓  
python import_incremental.py   
\--input-dir ./output   
\--metadata metadata.csv   
\--collection material_documents_qwen3_full

新增文件

  1. 02_ingestion/import_incremental.py — 增量灌库脚本（核心）


输入：

  * \--input-dir: MinerU 输出根目录，包含多个 /auto/_content_list.json
  * \--metadata: CSV 文件，包含每个 PDF 的元数据
  * \--collection: 目标 Milvus collection（默认 MATERIAL_COLLECTION_NAME）
  * \--gpu: GPU 编号（默认 0）
  * \--dry-run: 只解析不入库
  * \--batch-size: 文档批大小（默认 50）


metadata.csv 格式：  
filename,material_code,material_name,machine_type,specification,category  
AVAS-喇叭音腔.pdf,04.AV.1AV00123,AVAS喇叭,AV-00123,音腔规格,喇叭  
控制板SOP.pdf,04.AJ.1AJ0115GS0,控制板,AJ-0115-GS,,

  * filename 列：PDF 文件名（不含路径），用于匹配 MinerU 输出目录名（即 pdf_stem）
  * 其余列为 Milvus 元数据字段，可留空


核心逻辑（复用 V2 的处理管线）：

Phase 0: 扫描 + 元数据匹配

  * 递归扫描 input-dir 下所有 *_content_list.json
  * 从路径推导 pdf_stem（如 output/AVAS-喇叭音腔/auto/AVAS-喇叭音腔_content_list.json → pdf_stem = "AVAS-喇叭音腔"）
  * 用 pdf_stem 匹配 metadata.csv 的 filename 列（去掉 .pdf 后缀比对）
  * 未命中的文档隔离到 quarantine


Phase 1: 分块（复用 V2 逻辑）

  * load_content_list() → 读 JSON
  * group_blocks_by_page() → 按页分组 + bbox 排序
  * 每页：text 块合并 → chunk_text_by_tokens() 切分  
table 块 → build_table_chunk() 整表保留
  * in-batch content_hash 去重


Phase 2: 向量化

  * VectorEmbedderV2.encode_texts_safe() 批量编码
  * 校验维度(2560) + 零向量拒绝


Phase 3: 幂等写入

  * 对每个有 valid chunks 的 doc_address，先 DELETE 旧数据
  * flush() 确保删除落盘
  * FlexibleMilvusManagerV2.insert_chunks_v2() 插入新数据
  * flush_and_load()


从 V2 脚本复用的方法（直接 import 或复制）：

  * load_content_list() (line 484)
  * group_blocks_by_page() + _bbox_sort_key() (line 489/502)
  * merge_text_blocks_in_page() (line 515)
  * chunk_text_by_tokens() (line 525)
  * build_table_chunk() (line 622)
  * _build_chunk() (line 726)
  * is_boilerplate() + _has_valid_short_text_pattern() (line 409/460)
  * _delete_doc_chunks() (line 1256)
  * _truncate_by_bytes() (line 834)
  * _html_to_text() (line 719)


新增/替换的逻辑：

  * _scan_input_dir(input_dir): 递归扫描找 content_list.json，返回 [{path, pdf_stem, base_dir}]
  * _load_metadata_csv(csv_path): 读 CSV，按 filename 建索引 {pdf_stem: metadata_dict}
  * _resolve_metadata(pdf_stem): 用 pdf_stem 匹配 CSV 索引，返回元数据
  * process_single_doc(json_path, base_dir, metadata): 替代 V2 的 process_document()，直接接受路径+元数据

验证方案

  1. 单元验证：用 --dry-run 模式跑少量文档，检查分块结果是否正确
  2. 小批量验证：先灌 2-3 个文档，检查 Milvus 中数据是否正确
  3. 查询验证：灌库后用 API 查询，确认能检索到新文档内容
  4. 幂等验证：跑两次同一批文档，确认不产生重复数据
  5. 批量验证：全量跑，检查统计信息（processed / indexed / quarantined / failed）
