## 二、关键进展

### 1\. 完成 Agent 联网搜索方案接入

本周重点围绕组件异常问答 Agent 的联网检索链路进行优化，主要目标是替换原先不稳定的搜索引擎 HTML 爬取方案。

当前搜索层采用自部署 **SearXNG** 作为主搜索入口。

  


Agent 通过 SearXNG 的 `/search` 接口发起查询，优先使用 `format=json` 获取结构化结果；如果 JSON 接口异常或无结果，则回退解析 SearXNG HTML 页面，保证搜索链路具备一定容错能力。

相比直接爬取 DuckDuckGo、Bing、Yahoo 等搜索页面，当前方案主要有以下改进：

  * 不依赖商业搜索 API Key，降低 Tavily、Brave、Bing、SerpAPI 等 Key 缺失或过期带来的风险；
  * 优先使用结构化 JSON 结果，避免直接解析搜索引擎 HTML 页面；
  * 减少搜索引擎页面结构变化、反爬和超时问题对 Agent 的影响；
  * 支持统一的结果清洗、去重和相关性过滤；
  * 可以减少低价值页面、站内搜索页、商品页、地图页等干扰结果；


目前 SearXNG 搜索链路已完成基础接入，可以作为 Agent 联网检索的主入口。

* * *

### 2\. 完成 Web Fetch 多后端网页正文抓取方案优化

本周对搜索结果后的网页正文抓取能力进行了重点优化。当前 Agent 不再只依赖搜索结果中的 snippet 摘要，而是会对候选 URL 继续执行正文读取，获取更完整的网页证据。

当前网页读取层采用多后端容错方案，主要包括：

  * **Jina Reader** ：适合快速获取网页 Markdown 内容；
  * **Crawl4AI** ：适合动态网页、技术文章和论坛内容提取；
  * **Scrapling** ：用于增强动态页面抓取，对部分论坛页效果更好；
  * **Turndown** ：通过 Node.js 将 HTML 转换为 Markdown，作为补充后端；
  * **requests + BeautifulSoup** ：作为轻量兜底方案；
  * **snippet fallback** ：对于 PDF、已知慢站点或抓取失败页面，保留搜索摘要，避免证据链完全为空。


目前默认启用了 **competitive read** 模式，即并行运行多个 fetch 后端，并根据正文质量选择最优结果。

选择依据主要包括：

  * 正文长度；
  * 内容质量分；
  * 噪声比例；
  * 技术关键词覆盖情况；
  * 与查询上下文的匹配度；
  * 抓取耗时；
  * 后端失败原因。


这种方式可以避免单一抓取器在论坛页面、动态网页、编码异常页面上失效，提高整体网页证据获取的稳定性。

### 3\. 完成搜索与网页抓取链路测试

本周使用 `scripts/test_search_fetch_flow.py` 对搜索与抓取链路进行了多组测试。

测试内容主要包括：

  * 单样本搜索测试：验证 SearXNG 能否返回候选 URL；
  * 多样本搜索 + fetch 测试：验证搜索结果能否继续读取网页正文；
  * 直接 URL 抓取测试：验证 web fetch 后端在 EEWorld 等论坛页面上的表现；
  * 后端对比测试：对 Jina、Crawl4AI、Scrapling、Turndown 的正文质量、字符数、耗时和失败原因进行对比。


测试中观察到：

  * SearXNG 搜索链路可以替代原先不稳定的搜索引擎 HTML 爬取方式；
  * web fetch 能够从论坛、技术博客等页面中提取正文内容，不再只停留在搜索摘要；
  * 对 EEWorld 论坛页的直接抓取测试中，competitive read 可以返回 `FETCH_OK`；
  * 多后端对比后，可以选择正文质量更高的抓取结果；
  * 对于 datasheet、论坛问答、技术博客、电路问题分析等场景，新方案能提供更完整的证据内容。


目前搜索处理链路已经形成：
```text
SearXNG 搜索候选 URL
→ web_read 多后端抓取正文
→ 内容清洗
→ 质量评分
→ 选择最优网页证据
```

相比之前只依赖搜索 snippet 的方案，新链路能够为 Agent 提供更完整、更可追溯的网页证据。

下周计划基于最新一年的 PDF 数据进行增量灌库，但目前相关数据公司尚未提供。

* * *

### 2\. PDF 解析和知识库入库依赖 GPU

知识库入库过程中涉及 PDF 解析、版面识别、图片/表格处理、向量化等步骤，其中部分流程依赖 GPU。

如果目标环境没有可用显卡，将无法直接在目标环境完成完整解析和灌库流程。届时只能采用以下方式：
```text
自有服务器完成 PDF 解析、向量化和数据库构建
→ 导出或迁移库数据
→ 拷贝到目标环境使用
```

这种方式可行，但成本较高，主要风险包括：

  * 数据拷贝量较大；
  * 环境不一致导致问题复现困难；
  * 数据同步流程较重；
  * 后续更新维护不方便；
  * 问题排查链路变长。
