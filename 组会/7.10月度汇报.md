### 3. Async Context Compression 插件（6月）

针对 WebUI 平台开发的长对话上下文压缩插件，现已上线稳定运行。

#### 核心技术实现

**① 三级 Token 估算优化**

- **Fast 估算**：`_estimate_text_tokens()` ~0.01ms，误差±15%
- **Cached**：`@lru_cache`，O(1) 查表返回
- **Precise**：`tiktoken.o200k_base` ~0.1ms

设计效果：~80% 的场景只用 Fast 路径就能判断 Token 远低于阈值

**② 系统消息绝对保护**

三层保护确保 LLM 行为指令永不丢失：
- `keep_first` 只计数非系统消息 → 自动保留
- Gap 系统消息提取 → 以原始消息形式保留
- 强制裁剪兜底 → 提取并重新插入

**③ 原生工具输出裁剪**

- 工具输出字符数 > 600 则裁剪
- 替换为 `[Content collapsed]`，标记 `is_trimmed = True`

**④ 异步摘要生成 — 永不阻塞用户**

```
用户请求 → Inlet(~1ms) → LLM → 返回用户 → 后台 asyncio.create_task()
                              → Token 计算 → 生成摘要 → 保存数据库
```

**⑤ 摘要 Prompt 工程**

XML 结构化输出，包含 9 个预定义章节：

```xml
<working_memory>
  <current_goal>...</current_goal>
  <user_preferences>...</user_preferences>
  <persistent_context>...</persistent_context>
  <recent_progress>...</recent_progress>
  <tool_state>...</tool_state>
  <errors_and_warnings>...</errors_and_warnings>
  <open_loops>...</open_loops>
  <next_reply_guidance>...</next_reply_guidance>
</working_memory>
```

### 4. 知识库图片召回（ 本周完成）

#### 问题背景

当前知识库只能检索文本 chunk，无法返回对应的图片、截图、图纸。用户询问"查电芯装配流程图"时，只能返回文字描述，无法看到实际图片。

#### 4.1 技术选型

调研了三套方案并完成选型：

| 方案                       | 特点              | 选型结果  |
| ------------------------ | --------------- | ----- |
| 直接存储 Base64              | 实现简单、大幅增加 DB 体积 | ❌ 不采用 |
| MinIO 对象存储 + Milvus 向量检索 | 云原生架构、成熟稳定      | ✅ 推荐  |
| 本地文件 + Nginx 托管 + Milvus | 轻量简单、无额外依赖      | ✅ 采用  |

#### 4.2 方案概述

- **Embedding 后端**：支持 `sentence_transformers`（真实验证）和 `hash`（快速测试）
- **向量库**：Milvus 存储图片描述/元信息的 embedding 向量
- **文件托管**：FastAPI/Nginx 静态服务或 MinIO 对象存储

#### 4.3 项目架构

```text
data/
├── incoming/     # 待入库原始数据（业务方持续追加）
├── assets/       # FastAPI/Nginx/MinIO 托管图片
├── metadata/    # 归档后的描述文件
└── manifests/   # 增量状态和失败记录
```

#### 4.4 入库流程

1. 读取 `incoming` 目录下的图片 + 描述文件
2. 使用 BGE 模型生成文本 embedding
3. 提取图片元信息（image_hash、metadata_hash、content_hash）
4. 双层去重：
   - 第一层：检查 manifest 文件（`asset_type + source_id + page_no + content_hash`）
   - 第二层：查询 Milvus 主键（避免 manifest 丢失导致重复）
5. 写入 Milvus 并归档到 `assets`、`metadata`、`manifests`

#### 4.5 API 接口

| 接口 | 方法 | 功能 |
|------|------|------|
| `/health` | GET | 检查服务和 Milvus |
| `/stats` | GET | 返回 collection 统计 |
| `/search` | POST | 向量语义检索，返回 image_url 和业务字段 |
| `/assets/*` | GET | 静态图片服务 |

**检索示例：**

```bash
curl -s http://localhost:8020/search \
  -H 'Content-Type: application/json' \
  -d '{"query":"TM1640 LED驱动 3.3V转5V通讯","top_k":5,"asset_type":"schematic","score_threshold":0.35}'
```

**返回示例：**

```json
{
  "results": [{
    "score": 0.86,
    "asset_id": "schematic:CHM-10-30-001:p001:xxxx",
    "asset_type": "schematic",
    "source_id": "CHM-10-30-001",
    "title": "TM1640数码管/LED显示驱动模块",
    "summary": "基于TM1640-SOP28的LED显示驱动电路...",
    "category": "输出控制类",
    "image_url": "http://localhost:8020/assets/schematic/CHM-10-30-001/p001.png"
  }]
}
```

#### 4.6 验证结果

已通过 7 条原理图模块的检索验证，支持精确召回：

- `CHM-10-30-012` - 888LED排列数码 MCU大电流口扫描驱动
- `CHM-10-30-024` - LED状态指示 MCU大电流口扫描驱动 RGB脚位
- `CHM-10-30-031` - 数码管带按键扫描驱动 4位数码管 key扫描
- `CHM-10-30-041` - RGB三色AQI指示驱动 空气质量等级指示灯 PWM控制
- `CHM-10-30-051` - LED高速扫描 变亮度 呼吸效果 多路扫描
- `CHM-10-30-066` - LED PWM调亮度 背光 BackLight 调光
- `CHM-10-30-071` - PWM驱动器 HTR调光方案 36路18路 LED状态指示

同时支持无关内容过滤（`score_threshold` 参数），避免无关查询返回噪声结果。
