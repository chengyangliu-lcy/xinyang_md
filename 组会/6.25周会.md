## ✨ 一、关键进展

### 🔌 Async Context Compression — WebUI 上下文压缩 Filter 插件

> 针对 WebUI 平台开发的长对话上下文压缩插件。通过智能摘要和消息压缩降低 Token 消耗，同时保持多轮对话连贯性。

---

### 🏗️ 架构总览

**整体流程：**

- 用户发送消息 → Filter Pipeline → 加载历史摘要 → 组装压缩上下文 → 发送给 LLM → LLM 生成响应 → 响应后处理 → 异步计算 Token → 判断是否触发压缩 → 后台生成摘要 → 保存到数据库

**共享数据库引擎发现路径：**

1. 尝试 get_db_context
2. 失败则尝试 db_module.engine
3. 失败则尝试 ENGINE / BIND 等属性
4. 都失败则用 sessionmaker 兜底
5. 创建 chat_summary 表（如不存在）
6. 发现 schema 名称（多租户环境适配）

**数据库表结构：**

| 字段 | 类型 | 说明 |
|:---|:---|:---|
| `id` | INTEGER PK | 自增主键 |
| `chat_id` | VARCHAR(255) UNIQUE | 聊天会话 ID（唯一索引） |
| `summary` | TEXT | 摘要内容 |
| `compressed_message_count` | INTEGER | 已压缩到的原始消息数 |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

### ⚙️ 核心技术实现

---

#### 1. 三级 Token 估算优化

避免每轮对话都调用 tiktoken（有 Python 调用开销），设计了三级估算路径：

- **① Fast 估算**：_estimate_text_tokens() ~0.01ms，误差±15%
- **② Cached**：@lru_cache，O(1) 查表返回
- **③ Precise**：tiktoken.o200k_base ~0.1ms

**Fast 估算原理（基于字符特征，无需 tokenizer）：**

| 文本类型 | 估算策略 |
| :--- | :--- |
| **纯 ASCII**（英文/代码） | 按字符数 × 系数 0.17~0.20，额外考虑空格、标点和换行 |
| **含非 ASCII**（中文/日文等） | 采样前 256 个非 ASCII 字符，识别文字类型分布，按预设系数加权 |
| **文字类型系数** | 汉字 0.295 / 假名 0.235 / 谚文 0.175 / 西里尔 0.13 / 阿拉伯 0.135 / 泰文 0.145 |

**设计效果：** ~80% 的场景只用 Fast 路径就能判断 Token 远低于阈值，避免 tiktoken 开销。接近阈值时才降级精确计算。

---

#### 2. 系统消息绝对保护

系统消息是 LLM 行为指令的核心载体，三层保护确保永不丢失：

```
消息序列:  [sysA] [user1] [sysB] [user2] [user3] ... [userN-1] [userN]
               ↑                                      ↑            ↑
          keep_first=2                         保留结尾(keep_last=2)
          非系统消息计数: user1, user2
```

- **① keep_first 只计数非系统消息**：返回 index=4，包含 [sysA, user1, sysB, user2]（sys 自动保留）
- **② Gap 系统消息提取**：[user3]...[userN-3] 之间的 sys 消息 → 以原始消息形式保留
- **③ 强制裁剪兜底**：丢弃原子组时，组内的 sys 消息 → 提取并重新插入最终输出

**最终输出：** [sysA] [user1] [sysB] [user2] + [摘要] + [保留的Gap sys] + [userN-1] [userN]

---

#### 3. 原生工具输出裁剪

- 遍历原子分组，判断工具输出字符数是否 > 600
- 若是：tool content → '[Content collapsed]'，assistant 前追加 '[Tool outputs trimmed]'，标记 metadata is_trimmed = True
- 若否：保留完整内容

---

#### 4. 异步摘要生成 — 永不阻塞用户

**流程：**
1. 用户请求 → 插件 Inlet（~1ms）→ LLM 请求
2. LLM 响应 → 插件返回给用户（此时用户无感知）
3. 后台任务：asyncio.create_task() → Token 计算 → 生成摘要 → 保存到数据库

---

#### 5. 摘要 Prompt 工程

摘要 Prompt 设计为 **XML 结构化输出**，包含 9 个预定义章节：

```xml
<working_memory>
  <current_goal>...</current_goal>
  <user_preferences>...</user_preferences>
  <persistent_context>...</persistent_context>     ← 保守使用
  <recent_progress>...</recent_progress>           ← 优先放这里
  <tool_state>...</tool_state>
  <errors_and_warnings>...</errors_and_warnings>
  <open_loops>...</open_loops>
  <next_reply_guidance>...</next_reply_guidance>
</working_memory>
```
![[file-20260624163516438.png]]
**三种压缩风格对比：**

| 风格 | Token 消耗 | 适用场景 |
| :--- | :--- | :--- |
| `aggressive` | 最低 | 极长对话、低成本推理模型 |
| `balanced`（默认） | 中等 | 通用场景 |
| `faithful` | 较高 | 复杂推理、多项备选方案评估 |

**Prompt 元指令要点：**
- 旧摘要通过 `<previous_working_memory>` 传入 → **合并更新**而非替换
- `<persistent_context>` 遵循**保守原则**（不确定时放 `<recent_progress>`）
- 区分"已验证事实" vs "推断结论" vs "开放问题"
- 代码、路径、配置值、错误字符串要求**逐字保留**

### 🔧 关键配置

| 参数 | 默认值 | 说明 |
| :--- | :--- | :--- |
| `compression_threshold_tokens` | 64000 | 触发压缩的 Token 阈值 |
| `max_context_tokens` | 128000 | 上下文硬限制 |
| `model_thresholds` | "" | 按模型单独设阈值，如 `gpt-4:8000:32000` |
| `keep_first` | 0 | 保留前 N 条非系统消息 |
| `keep_last` | 6 | 保留后 N 条完整消息 |
| `compression_style` | balanced | 压缩风格 |
| `summary_model` | None | 指定摘要模型 |
| `enable_tool_output_trimming` | true | 工具输出裁剪 |
| `tool_trim_threshold_chars` | 600 | 裁剪阈值（字符） |
| `summary_fail_mode` | silent | 摘要失败：silent / raise |
