### 3. Async Context Compression 插件（6月）

针对 WebUI 平台开发的长对话上下文压缩插件，现已上线稳定运行。

#### 核心技术实现

**① 三级 Token 估算优化**

- **Fast 估算**：`_estimate_text_tokens()` ~0.01ms，误差±15%
- **Cached**：`@lru_cache`，O(1) 查表返回
- **Precise**：`tiktoken.o200k_base` ~0.1ms

设计效果：~80% 的场景只用 Fast 路径就能判断 Token 远低于阈值

**② 系统消息绝对保护**

三层保护确保 LLM 行为指令永不丢失：
- `keep_first` 只计数非系统消息 → 自动保留
- Gap 系统消息提取 → 以原始消息形式保留
- 强制裁剪兜底 → 提取并重新插入

**③ 原生工具输出裁剪**

- 工具输出字符数 > 600 则裁剪
- 替换为 `[Content collapsed]`，标记 `is_trimmed = True`

**④ 异步摘要生成 — 永不阻塞用户**

```
用户请求 → Inlet(~1ms) → LLM → 返回用户 → 后台 asyncio.create_task()
                              → Token 计算 → 生成摘要 → 保存数据库
```

**⑤ 摘要 Prompt 工程**

XML 结构化输出，包含 9 个预定义章节：

```xml
<working_memory>
  <current_goal>...</current_goal>
  <user_preferences>...</user_preferences>
  <persistent_context>...</persistent_context>
  <recent_progress>...</recent_progress>
  <tool_state>...</tool_state>
  <errors_and_warnings>...</errors_and_warnings>
  <open_loops>...</open_loops>
  <next_reply_guidance>...</next_reply_guidance>
</working_memory>
```

### 4. 知识库图片召回方案（本周新进度）

针对现有知识库无法召回图片的问题，设计了三套技术方案。

#### 问题背景

当前知识库只能检索文本 chunk，无法返回对应的图片、截图、图纸。用户询问"查电芯装配流程图"时，只能返回文字描述，无法看到实际图片。

#### 方案一：文本检索 + 页面截图返回（最快上线）

**特点**：复用现有文本检索，根据命中结果返回对应页面截图。

**架构**：
```
文档入库阶段：
PDF → 文本切片 + embedding → Milvus 入库
     → 每页渲染 PNG → 保存 page_no → page_image_url 映射

查询阶段：
OpenWebUI Tool → /search/dify 文本检索
     → 返回 doc_address + page_no
     → 后端补充 page_image_url
     → OpenWebUI 展示文本 + 页面截图
```

**目录结构**：
```
/data/xinyang_knowledge/
  pdf/
    doc123.pdf
  pdf_pages/
    doc123/
      page_0001.png
      page_0002.png
  page_manifest/
    doc123.json
```

**返回格式**：
```json
{
  "results": [
    {
      "result_type": "text_with_page_image",
      "score": 0.83,
      "doc_name": "电芯装配SOP.pdf",
      "page_no": 7,
      "content": "电芯装配流程包括上料、扫码、焊接...",
      "page_image_url": "http://10.1.120.36:5010/pdf_pages/doc123/page_0007.png",
      "neighbor_page_image_urls": [
        "http://10.1.120.36:5010/pdf_pages/doc123/page_0006.png",
        "http://10.1.120.36:5010/pdf_pages/doc123/page_0007.png",
        "http://10.1.120.36:5010/pdf_pages/doc123/page_0008.png"
      ]
    }
  ]
}
```

**优点**：改动最小，最快上线；不需要升级 Milvus；不需要图像模型；用户看得到图

**缺点**：不是真正的视觉检索；依赖文本检索准确性；需要入库时保留 page_no

#### 方案二：OCR/Caption 转文本 + 文本检索

**特点**：把图片转成文字描述（caption），写入原有文本知识库，检索 caption 时返回对应图片 URL。

**架构**：
```
图片 / 页面截图
  → 保存到静态文件服务（得到 image_url）
  → VLM 生成 caption / OCR 提取文字
  → caption 做文本 embedding
  → caption + image_url + doc_id + page_no 一起写入 Milvus
  → 用 /search/dify 检索 caption，返回时带上 image_url
```

**核心原则**：图片 URL 在入库时生成，而不是检索时凭空得到的

**Milvus Schema**：
```python
{
  "id": "doc123_p0007_img0001",
  "content": "这是一张电芯装配流程图，展示上料、扫码、焊接、检测、下料等步骤。",
  "embedding": [0.01, 0.02, "..."],
  "image_url": "/kb-images/doc123/doc123_p0007_img0001.png",
  "doc_address": "/pdf/doc123.pdf",
  "doc_name": "电芯装配SOP.pdf",
  "doc_id": "doc123",
  "page_no": 7,
  "category": "流程图",
  "content_type": "image_caption"
}
```

**推荐接口**：`/search/image_caption`

**优点**：实现简单；旧文本检索完全复用；可搜性提升

**缺点**：Caption 质量不稳定；视觉结构信息丢失；不是真正的视觉检索

---

#### 方案三：ColQwen3 多向量视觉检索（推荐进阶）

**特点**：真正的视觉语义检索，按图像内容召回，不依赖文字。

**架构**：
```
OpenWebUI Tool
  → /search/multimodal
  → tomoro-colqwen3-embed-4b 编码 query
  → Milvus 2.5.14 搜 patch_vec
  → 按 page_id 聚合候选页面
  → 从本地/数据库取完整 patch embeddings
  → 应用层计算 MaxSim rerank
  → 返回 page_image_url / doc_address / page_no / score
```

**Collection 设计**（`visual_page_patches`）：
| 字段 | 类型 | 说明 |
|------|------|------|
| patch_id | VARCHAR | 主键，如 `doc123_p0007_t0001` |
| page_id | VARCHAR | 页面 ID，如 `doc123_p0007` |
| doc_id | VARCHAR | 文档 ID |
| patch_vec | FLOAT_VECTOR(320) | patch 向量 |
| page_image_url | VARCHAR | 页面截图 URL |
| doc_address | VARCHAR | 原 PDF 地址 |

**查询流程**：
1. query → ColQwen3: [q_len, 320]
2. 每个 query token 去 Milvus 搜 top_n patches
3. 合并命中 page_id，得候选页面
4. 读候选页完整 patch embeddings
5. MaxSim: score(query, page) = sum_i max_j dot(q_i, p_j)
6. 排序返回 top_k 页面

**优点**：能按视觉内容、结构、版面、图表召回；检索质量最高；与旧文本检索隔离

**缺点**：工程复杂，需多向量检索 + 应用层 rerank；存储膨胀；需要升级 Milvus 2.6

**适用场景**：视觉结构搜索，如"左上角有蓝色框的流程图"、"包含三条曲线的实验图"

---

#### 三套方案对比

| 对比项 | 方案一：文本+截图 | 方案二：OCR/Caption | 方案三：ColQwen3 |
|--------|-----------------|---------------------|------------------|
| **工程复杂度** | 低 ⭐ | 中 ⭐⭐ | 高 ⭐⭐⭐ |
| **上线速度** | 快 | 中 | 慢 |
| **本质** | 文本命中→返回截图 | 把图翻译成文字再搜 | 直接搜图片本身 |
| **检索对象** | 文本 chunk | 文字描述 caption | 视觉向量 |
| **入库成本** | 低，PDF渲染 | 中，VLM生成caption | 高，多向量编码 |
| **检索成本** | 低 | 低 | 高，MaxSim rerank |
| **需要升级 Milvus** | 否 | 否 | 是 |
| **需要 GPU** | 否 | 入库时需要 | 编解码都需要 |
| **适合场景** | 任何带文本的文档 | 简单图/流程图/照片 | 复杂图纸/扫描件/表格 |
| **推荐阶段** | 第一阶段快速上线 | 第二阶段补充 | 第三阶段增强 |

**一句话总结**：方案一最简单，方案二次之，方案三最难

#### 技术选型建议

| 阶段 | 推荐方案 | 理由 |
|------|----------|------|
| **第一阶段** | 方案一：文本+截图 | 最快上线，改动最小，复用现有文本检索 |
| **第二阶段** | 方案二：OCR/Caption | 补充图片可搜性，入库时一次性处理 |
| **第三阶段** | 方案三：ColQwen3 | 进阶视觉检索，处理复杂图纸/扫描件 |
