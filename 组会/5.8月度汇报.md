## Slide 2: 项目概述

​  


**项目定位**:

面向电子工程领域的 AI Agent 问答系统。输入中文技术问题 + 电路板图片，Agent 自主调用知识库检索、网络搜索、图片分析等工具，收集证据并生成结构化技术答案。

​  


**核心能力**:

\- 自主工具调用 (Agentic Tool Loop)

\- 本地知识库混合检索 (Dense + Sparse + Cross-Encoder Rerank)

\- 网络证据采集 (多搜索引擎 + 网页阅读)

\- 多模态图片分析 (电路板/原理图)

\- 多维度自动评估 (LLM Judge + 结构化评分点 + 语义相似度)

​  


**技术栈**: Python 3.12+ · Qwen3.6-Plus (DashScope) · SQLite FTS5 · Qwen3-Embedding-4B · BGE-Reranker-v2-M3 · BERTScore

## KB 构建流程详解 (2) — 三阶段构建管道

​  


**Phase 1: 快速扫描 + 元数据过滤** (单线程, 低开销)

```

遍历所有 page_*.md 文件

→ 仅读取文件头 8KB (parse_markdown_metadata)

→ 检查标题合法性 (拒绝 404/空白/乱码)

→ 检查 URL 路径白名单/黑名单

→ 检查文件大小 (拒绝 > 2MB)

→ 检查诊断关键词 (无诊断标记的页面直接丢弃)

→ 收集候选 → 分批送入 Phase 2

```

​  


**Phase 2: 并行清洗 + 过滤** (ProcessPoolExecutor, 8 workers)

```

对每个候选文件:

→ 完整读取 Markdown (parse_markdown_page)

→ 文本清洗 (clean_public_tech_text)

├─ 定位文章正文起始 (标题检测 / 连续段落检测)

├─ 切除尾部噪音 (相关推荐/热门文章/评论区/nav)

├─ 过滤行内噪音 (广告/登录/SNS/打赏/版权)

├─ 清除图片 Markdown/Base64/CDN URL

└─ 切除尾部分页索引/标签行

→ 质量校验 (is_useful_page)

├─ 文本长度 ≥ 160 字符

├─ 字母/CJK 字符占比 ≥ 35%

└─ 电子术语命中数 ≥ 2

```

​  


**Phase 3: 分块 + 入库** (主线程, SQLite WAL 模式)

```

清洗后文本 → chunk_text (900 字符/chunk, 120 字符重叠)

→ 在句号/换行处断句 (智能边界检测)

→ 去重 (URL + text_hash 双重去重)

→ 写入 SQLite: pages 表 + chunks 表 + FTS5 全文索引

```

​  


**并发设计**: Phase 1 和 Phase 2 交错执行 (batch size=500~5000)，边扫描边处理，避免等待全量扫描完成。

## 构建流程详解 (3) — 文本清洗策略

​  


**核心挑战**: 网络文章混有大量非技术内容 (导航栏、广告、评论区、推荐列表)

​  


**正文定位** (两阶段检测):

```

策略 1: 找第一个干净的 # 或 ## 标题

→ 排除噪音标题词: 404, 技术专区, logo, 热门推荐...

→ 要求标题 ≥ 6 字符

​  


策略 2: 找连续 3 行以上的正文段落

→ 行长度 ≥ 40 字符

→ 字母/CJK 字符占比 > 50%

```

​  


**尾部噪音切除** (4 层防护):

```

1\. 中文段落停止词 (35+ 种):

"相关技术文章", "相关资料下载", "用户评论",

"热门推荐", "论坛热帖", "深度阅读"...

​  


2\. 英文/俄语段落停止词 (20+ 种):

"related articles", "most popular", "comments",

"previous article", "similar projects"...

​  


3\. 中文后缀标记 (50+ 种):

"技术专区", "相关帖子", "版权申明", "扫码关注"...

​  


4\. 尾部分页索引检测 (自动识别):

连续 3+ 行 "数字. 标题" 格式的行 → 切除

```

​  


**行内噪音过滤** (100+ 种标记):

```

过滤: 广告/SNS/登录/打赏/注册/积分/下载信息/面包屑导航/

元器件搜索框/订阅/分享/打印/收藏/版权/二维码...

```

## KB 构建流程详解 (4) — 检索管道

​  


**检索全链路** (从用户 Query 到最终 Evidence):

​  


```

Query: "TL431反馈环路振荡怎么排查？"

│

▼

┌─ Step 1: 查询预处理 ───────────────────────┐

│ extract_query_terms(query) │

│ → 拆解: [TL431, 反馈, 环路, 振荡, 排查] │

│ → 中英扩展 (ZH_EN_QUERY_EXPANSIONS): │

│ 反馈→feedback, 振荡→oscillation, │

│ TL431 识别为 model term │

│ → 术语分类: models/TL431, topology/feedback│

│ fault/oscillation, other/排查 │

│ │

│ LLM Query Translation (enable_llm_query_ │

│ rewriting=true): │

│ "TL431反馈环路振荡排查" → LLM → │

│ "TL431 feedback loop oscillation │

│ troubleshooting compensation" │

└─────────────────────────────────────────────┘

│

▼

┌─ Step 2: 稀疏检索 (SQLite FTS5) ───────────┐

│ build_fts_query(query): │

│ 优先匹配强信号词 (models/refdes/values): │

│ FTS5: "TL431" OR ... │

│ 候选上限: 80 条 (candidate_limit) │

│ BM25 排序 + rerank_rows 二次评分 │

│ → 标题命中 +3.0, 正文命中 +1.0 │

│ → model term +5.0/-1.5, refdes +2.0 │

│ → topology +2.0, fault +1.2 │

│ → high_value_marker +0.8 │

│ → boilerplate -8.0, noise -3.0 │

└─────────────────────────────────────────────┘

│

▼

┌─ Step 3: 稠密检索 (Dense) ─────────────────┐

│ DenseRetriever.search(query, limit=80): │

│ Qwen3-Embedding-4B → 2560-dim向量 │

│ NumPy 内积搜索 → Top-80 chunk_id │

│ 返回 (余弦相似度, chunk_id) 列表 │

└─────────────────────────────────────────────┘

│

▼

┌─ Step 4: 混合融合 ─────────────────────────┐

│ 对每个候选 chunk_id: │

│ sparse_norm = sparse_score / max_sparse │

│ dense_norm = dense_score (0-1) │

│ hybrid = 0.65*dense_norm + 0.35*sparse │

│ │

│ Dense-only 补充 (dense≥0.3 但未命中稀疏): │

│ 按 0.65*dense_norm 加入候选 │

└─────────────────────────────────────────────┘

│

▼

┌─ Step 5: Cross-Encoder 精排 (可选) ────────┐

│ BGE-Reranker-v2-M3: │

│ 输入 (query, chunk_text) pairs │

│ Top-30 候选 → Cross-Encoder 打分 │

│ 按 CE 分数重新排序 │

│ CE≥0.5 → high_relevance=true │

└─────────────────────────────────────────────┘

│

▼

┌─ Step 6: 质量过滤 + URL 去重 ──────────────┐

│ 依次过滤: │

│ ❌ low_value_source (circuitmaker用户页等) │

│ ❌ boilerplate_text (噪声>2处且无技术词) │

│ ❌ low_value_project (无强信号词的项目页) │

│ ❌ required_terms_mismatch (关键词未命中) │

│ ❌ 同 URL 超过 max_chunks_per_url (默认2) │

│ ❌ hybrid_score < threshold (默认0.15) │

│ │

│ 最终留存 top-4 Evidence → 送入 Agent 推理 │

└─────────────────────────────────────────────┘

```

## Slide 7: 技术亮点

​  


**1. Agentic Tool Loop 架构**

\- OpenAI/OpenHands 风格的 action/observation 循环

\- 8 个可插拔工具, 统一 ToolSpec schema

\- 参数自动修复: 缺失字段填充、查询重写、中英翻译

\- 预算控制: 每工具独立重试上限、全局超时 300s

​  


**2. 混合检索管道**

\- **Dense**: Qwen3-Embedding-4B (2560 维) → NumPy 内积搜索

\- **Sparse**: SQLite FTS5 trigram → BM25

\- **Fusion**: 加权融合 (dense 65% + sparse 35%) + 阈值过滤

\- **Rerank**: BGE-Reranker-v2-M3 Cross-Encoder 二次排序 (top-30)

\- **Query Rewrite**: LLM 将中文问题翻译为英文检索关键词

​  


**3. 多维度评估体系**

\- LLM Judge: 准确性、完整性、事实一致性、有用性、清晰度 (0.45)

\- 结构化评分点: 6 种评分点类型, 加权覆盖 + 矛盾检测 (0.25)

\- 语义相似度: BERTScore Chinese + sentence-transformers (0.20)

\- Claim ROUGE-L: 声明级最长公共子序列 (0.05)

\- 技术实体匹配: 元件 ID、型号、参数值 F-beta (0.05)

## Slide 9: 问题与挑战

​  


**1. KB 贡献度有限 (核心问题)**

\- KB 开关实验差异 < 0.03 分

\- KB 检索到的内容可能缺乏因果机制、公式、诊断步骤等关键信息

\- 即使正确检索到证据, LLM 未必有效利用 (证据→答案 gap)

​  


**2. 检索精度 vs 噪音的平衡**

\- 旧 KB (57k chunks) 覆盖广但噪音大

\- 新 KB (6k chunks) 精度高但可能覆盖不足

\- 当前策略偏向精度, 需要扩大高质量数据源

​  


**3. 评估体系的局限性**

\- LLM Judge 与人工评判的相关性未充分验证

\- KB 证据利用率指标 (kb_evidence_used_rate) 已损坏, 需修复

\- 缺少领域专家人工评审的对比基准

​  


**4. 工程效率**

\- 全部开发在 main 分支直接推送, 无分支协作

\- 实验管理依赖文件命名约定, 缺少实验追踪系统

\- 推理耗时长, 19 组实验完整运行需要大量 API 调用
