# ComponentAnomalyAgent 当前技术方法报告

ComponentAnomalyAgent 是面向电子组件异常分析问答任务的多模态 Agent 系统。

## 总体架构

系统由五层组成：

  1. 数据解析层：读取 `2025_dataset.jsonl`，解析 `sample_id/post_id`、题面、参考答案和图片路径。
  2. Agent 推理层：当前默认使用 `agentic_tool_loop`，即 OpenHands 风格的 action/observation 证据循环。
  3. 工具证据层：提供图片检查、领域技能匹配、联网搜索、网页读取、证据排序、证据审查和最终答案合成。
  4. 评测层：统一计算 LLM Judge、语义相似度、ROUGE-L、Bigram Jaccard、采分点覆盖率和综合分。
  5. 实验层：提供推理、评测、baseline、对比、去重、样本选择、数据校验等脚本。


整体数据流如下：
```text
数据集 JSONL
  -> 样本解析与图片路径解析
  -> Agentic Evidence Runtime
       -> planner 选择单个工具
       -> 工具执行并返回 observation/evidence
       -> 证据累计、去重、排序、审查
       -> finish_answer 合成最终答案
  -> predictions.jsonl + traces/*.trace.json
  -> 统一评测 eval_results.jsonl
  -> baseline 对比与实验报告
```

## 核心方法：Agentic Evidence Runtime

当前默认推理策略是 `agentic_tool_loop`。它不是一次性把问题丢给大模型，而是把每个样本变成一个多轮只读证据收集过程。每一轮 runtime 都维护当前状态，包括样本、图片路径、已收集证据、已选动作、搜索 query、已读 URL、工具调用次数、错误列表和停止原因。

### 3.1 Action/Observation 循环

每轮流程如下：

  1. planner 查看当前状态，包括题面、已有证据、预算、工具历史和错误。
  2. planner 只选择一个下一步动作，返回严格 JSON：


```json
{
  "tool_name": "inspect_image|match_domain_skill|web_search|web_read|rank_evidence|review_evidence|finish_answer",
  "args": {},
  "reason": "简短审计原因",
  "stop": false
}
```

  3. runtime 校验动作是否合法，检查预算、重复调用、搜索次数、图片检查次数、网页读取页数等约束。
  4. 对应工具执行并返回 observation，包括是否成功、摘要、证据、错误和是否可恢复。
  5. runtime 累计证据、去重、更新 trace 和工具计数。
  6. 如果证据足够或预算不足，则进入 `rank_evidence` 和 `finish_answer`。


该设计的核心价值是把“模型思考”外显为可审计的工具选择记录，同时限制每一轮只做一个动作，降低无边界推理和重复搜索的风险。

### 3.2 支持的工具动作

当前 runtime 支持七类动作：

动作| 目的| 典型使用场景  
---|---|---  
`inspect_image`| 使用视觉模型抽取图片中的元件、丝印、数值、连接、波形、异常痕迹| 有输入图片且尚未检查时优先使用  
`match_domain_skill`| 匹配电子领域通用技能和先验机制| 反馈环路、补偿、滤波、电源、接地、驱动、采样、选型问题  
`web_search`| 检索公开资料| 需要型号资料、数据手册、拓扑规则或公开原理证据  
`web_read`| 读取公开网页正文| 搜索摘要有价值且需要进一步确认  
`rank_evidence`| 根据题面关键词、来源质量、图片证据和领域技能排序去重| 收集到多来源证据后  
`review_evidence`| 检查证据覆盖度| 判断是否缺图片、元件、原因或处理建议  
`finish_answer`| 基于排序证据生成最终中文答案| 证据足够、预算压力或继续调用价值较低时  
  
所有工具均为只读工具，不修改文件或外部系统。

### 3.3 确定性 fallback 机制

为了避免 planner 不可用、模型返回非法 JSON 或 API 超时时整条样本失败，runtime 内置确定性 fallback action。fallback 顺序大致为：

  1. 如果有图片且未检查，先 `inspect_image`。
  2. 如果未匹配领域技能，执行 `match_domain_skill`。
  3. 如果允许联网且仍有 query 预算，执行 `web_search`。
  4. 如果有未读搜索结果且仍有网页读取预算，执行 `web_read`。
  5. 如果已有证据且未排序，执行 `rank_evidence`。
  6. 如果未做证据审查，执行 `review_evidence`。
  7. 最后执行 `finish_answer`。


这使系统具备“模型规划优先、规则兜底保障”的稳定性。即使 planner 短暂失败，系统仍能沿着合理证据路径继续推理。

### 3.4 证据足够性与强制收束

runtime 会在证据达到一定条件后强制推进到排序和答案合成。例如：

  1. 同时具备图片证据和领域技能证据。
  2. 已经获得网页证据。
  3. 证据数量达到较高水平。


满足条件后，如果尚未排序，则强制 `rank_evidence`；排序后强制 `finish_answer`。该机制避免 Agent 在已经有足够信息后继续无意义搜索，控制延迟和成本。

## 4\. 多模态图片处理方法

图片是电子组件异常任务中的关键输入。系统根据样本中的图片文件名解析本地路径，路径规则为：
```text
2025///images/
```

需要强调的是，系统只允许读取样本输入图片，不读取同目录下的 Markdown、HTML、JSON 作为知识证据，从而避免测试集泄漏。

图片处理由 `ImageInspectExecutor` 完成。其方法是：

  1. 找到存在的本地图片，最多发送配置中的 `max_llm_images` 张。
  2. 对图片做尺寸和质量控制，默认最长边 `1280`，JPEG 质量 `75`。
  3. 通过 OpenAI-compatible image payload 发送给视觉模型。
  4. 视觉提示词要求模型只描述可见内容，不给最终维修结论。
  5. 输出内容以 `image_inspection` 证据形式进入证据池。


视觉提示词重点要求覆盖：

  1. 整体画面：图片类型、电路板区域、仪表或波形。
  2. 元件与标注：元件位号、丝印、封装、数值和单位。
  3. 连接与拓扑：可见连接、供电、地、反馈、采样、驱动区域。
  4. 测量与异常线索：仪表读数、波形、烧蚀、断线、虚焊、污染等。
  5. 无法确认的信息：看不清的内容必须明确写“无法确认”。


## 5\. 领域技能与知识约束

系统支持从 `knowledge_base/domain_skills.yaml` 加载电子领域技能。领域技能用于补充通用工程知识，例如反馈环路、补偿、滤波、电源异常、驱动、采样和器件选型等。

领域技能匹配方式当前是关键词触发：问题文本命中某个技能的 triggers 后，系统将该技能作为 `domain_skill` 证据加入证据池。证据排序时，领域技能会获得额外权重，因为它通常能提供稳定的电子工程先验。

## 6\. 联网搜索与网页读取

系统支持多 provider 联网搜索，配置顺序在 `configs/default.yaml` 中：
```yaml
provider_order:
  - tavily
  - brave
  - bing
  - serpapi
  - html
```

如果 API key 缺失或 provider 失败，会继续尝试下一个 provider，最终可回退到 HTML 搜索。搜索结果会被转换为统一的 Evidence 对象，包含标题、URL、摘要、来源、分数和元数据。

网页读取由 `RobustWebReadExecutor` 完成，具备以下鲁棒性设计：

  1. 对 PDF、二进制、厂商慢站或容易被拦截页面，不强行读取正文，而是保留搜索摘要作为证据。
  2. 网页读取失败但搜索摘要存在时，仍将摘要作为可用证据返回，避免整条样本中断。
  3. 每个样本网页读取数量受配置限制，防止长时间抓取。
  4. 已读 URL 不会重复读取。

## 7\. 证据排序、去重与审查

收集到的证据不会直接全部塞给最终模型，而是先进行去重和排序。

证据去重依据包括：

  1. source。
  2. evidence kind。
  3. 压缩后的内容片段。


证据排序主要考虑：

  1. 题面关键词是否命中证据标题、正文或来源。
  2. 领域技能证据加权。
  3. 图片检查证据加权。
  4. 可信来源加权。
  5. 原始搜索或工具分数。


排序后最多保留配置数量的高价值证据。最终答案合成阶段还会按证据类型设定优先级：

  1. `image_inspection`
  2. `domain_skill`
  3. `web_page`
  4. `web_search_result`
  5. `image_context`


此外，`review_evidence` 会检查证据覆盖是否充分，尤其关注是否缺少图片、元件、原因或处理建议。若证据不足，最终答案提示词要求明确写出不确定性，不能把猜测写成确定事实。

## 8\. 最终答案生成策略

最终答案由 `FinishAnswerExecutor` 负责。它会把排序后的证据格式化为紧凑文本，再调用答案模型生成中文技术答案。

最终答案提示词有几个关键约束：

  1. 先直接回答问题，把最可能原因、计算结论或核心对比放在开头。
  2. 不要泛泛地说“检查供电、检查接地”，必须指向题面中的节点、元件、型号、数值或异常现象。
  3. 计算题必须给公式、代入值和选型步骤。
  4. 不得编造证据外的精确型号、参数、波形、测量值或因果关系。
  5. 图片里“疑似”“看不清”的连接关系不能升级为确定结论。
  6. 证据不足时说明缺口和复核项。
  7. 输出结构建议为：结论、依据、原因机制、检查步骤、处理建议与不确定性。


提示词还针对常见电子题型加入了专门规则：

题型| 必须覆盖的技术要点  
---|---  
充电芯片或 LED 状态异常| LED/状态输出逻辑、限流电阻、电流路径、漏电或高阻态、规格书核对、替换验证  
NTC/浪涌限流选型| R25、最大稳态电流、浪涌峰值、浪涌能量、220VAC 整流峰值约 311V、无 NTC 影响  
缓启动/预充电阻计算| 峰值功率、脉冲能量、平均功耗、RC 时间常数、相关电阻电容和 MOS 栅极  
运放恒流源/负反馈| 误差放大器闭环调节、MOS 栅极控制、采样电阻反馈电压、I=Vref/Rsense  
RC 或三极管振荡/闪烁灯| 起振条件、导通/截止顺序、电容充放电路径、阈值电压  
  
如果答案模型不可用，系统会使用确定性 fallback answer：基于领域技能、图片摘要和网页摘要生成保守答案，并明确证据不足。

## 9\. 配置与运行预算

系统通过 `configs/default.yaml` 集中管理路径、模型、Agent 行为、联网搜索、评测权重和 runtime 选项。

当前主要模型配置：

用途| 默认模型  
---|---  
Agent/planner/answer| `qwen3.6-plus`  
Vision| `qwen3.6-plus`  
Judge| `qwen-plus`  
  
当前主要预算参数：

参数| 默认值| 作用  
---|---|---  
`max_iterations`| 8| 每个样本最多 Agent 轮数  
`max_total_seconds`| 180| 每个样本总耗时上限  
`planner_timeout_seconds`| 25| planner 超时  
`tool_timeout_seconds`| 20| 普通工具超时  
`vision_timeout_seconds`| 35| 图片检查超时  
`final_answer_timeout_seconds`| 60| 最终答案生成超时  
`web_read_timeout_seconds`| 15| 网页读取超时  
`max_consecutive_tool_errors`| 3| 连续工具错误上限  
`max_planner_failures`| 2| planner 失败后熔断并使用 fallback  
`max_web_queries`| 3| 每样本最大搜索 query 数  
`max_web_pages_to_read`| 1 或 web 配置覆盖| 每样本最大网页读取数  
`max_llm_images`| 4| 每样本最多送入视觉模型的图片数  
`max_image_inspect_attempts`| 1| 图片检查最大尝试次数  
  
这些预算确保系统在批量实验时可控，不会因为单个样本的搜索、网页读取或模型调用卡住整体流程。

## 10\. Trace 与可解释性

每个样本推理会输出两类结果：

  1. `predictions.jsonl`：样本级预测结果。
  2. `traces/<sample_id>.trace.json`：详细工具调用轨迹。


预测结果字段包括：

字段| 含义  
---|---  
`sample_id`| 样本 ID  
`question`| 题面  
`answer`| 最终答案  
`tools_used`| 使用过的工具  
`web_searched`| 是否执行过联网搜索  
`tool_trace`| 工具事件列表  
`reasoning_summary`| 简短推理摘要  
`elapsed_seconds`| 推理耗时  
`token_usage`| token 使用记录  
`errors`| 非致命错误  
`plan`| 策略、预算、动作序列和停止原因  
  
trace 中会记录每个事件的工具名、动作名、耗时、输入、输出、是否成功、错误信息、证据变化量和剩余预算。这样可以回答三个关键问题：

  1. Agent 为什么会给出这个答案？
  2. 它基于哪些图片、技能或网页证据？
  3. 如果答错，是证据没收集到、证据排序错、还是最终合成错？


## 11\. 数据防泄漏设计

防止测试集泄漏是当前方法的重要约束。系统采取了多层控制：

  1. `2025/` 目录中的 Markdown、HTML、JSON 被视为测试集原始资料，默认禁止作为检索知识库。
  2. Agent 允许读取样本输入图片，但不能读取测试集原始帖子资料。
  3. 本地知识库默认路径是 `knowledge_base/`，且默认 `enable_local_retrieval: false`。
  4. 领域技能加载时禁止路径落在测试集目录或图片根目录。
  5. 联网搜索只使用公开网页证据，并在 trace 中记录来源。
  6. 评测使用参考答案，但参考答案只在评测阶段使用，不参与推理。


这一设计使推理阶段的证据来源可以被审计，也能在答辩或汇报中解释为什么实验没有利用测试集原始答案或帖子内容。

## 评测指标

**1\. Final Score，综合总分**

` Final Score` 是当前最主要的总评分，用来综合衡量答案质量。它不是单一指标，而是多个指标加权得到：
```text
FinalScore = 0.50 * LLMJudgeScore
           + 0.25 * SemanticSimilarity
           + 0.10 * ScoringPointCoverage
           + 0.10 * RougeL
           + 0.05 * BigramJaccard
```

**2\. LLM Judge Score，大模型裁判分**

` LLM Judge Score` 是由独立评测模型对预测答案进行质量判断，范围是 `0-1`。它会对比题目、参考答案、预测答案和采分点覆盖情况，综合判断答案质量。

内部主要看五个方面：
```text
LLMJudgeScore =
  0.35 * AccuracyNorm
+ 0.25 * CompletenessNorm
+ 0.20 * FactualConsistency
+ 0.10 * UsefulnessNorm
+ 0.10 * ClarityNorm
```

其中 `Accuracy`、`Completeness`、`Usefulness`、`Clarity` 原始分是 `1-5` 分，会先归一化到 `0-1`。

这个指标适合技术问答，因为它能识别“表达不同但技术含义正确”的答案，也能惩罚“看起来很完整但核心结论错了”的答案。

**2.1 Accuracy，准确性**

` Accuracy` 是 LLM Judge 给出的准确性评分，范围是 `1-5` 分。

它衡量预测答案的核心结论是否正确，例如：

  * 故障原因是否判断对了
  * 元件作用是否解释对了
  * 计算公式或参数关系是否正确
  * 是否把电路机制说反
  * 是否错误归因


如果答案命中了参考答案的核心技术结论，`Accuracy` 会高；如果核心原因错了，即使语言流畅，也会被扣分。

**2.2 Completeness，完整性**

` Completeness` 是完整性评分，范围是 `1-5` 分。

它衡量答案是否覆盖了参考答案中的主要要点，例如：

  * 是否说明了原因机制
  * 是否提到关键元件、节点、型号或参数
  * 是否给出必要检查步骤
  * 是否包含处理建议
  * 计算题是否给出公式、代入和结论


一个答案可能准确但不完整，比如只说“这是反馈补偿”，但没有解释相位补偿、高频滤波、稳定环路等内容，这种情况下 `Accuracy` 可能还可以，但 `Completeness` 会偏低。

**2.3 Clarity，清晰度**

` Clarity` 是表达清晰度评分，范围是 `1-5` 分。

它衡量答案是否结构清楚、语言顺畅、容易理解，例如：

  * 是否有明确结论
  * 是否分层说明
  * 是否逻辑顺序清楚
  * 是否避免冗长混乱
  * 是否没有自相矛盾


在你的实验里，Baseline 的 `Clarity` 高于 Agent，说明 baseline 的回答往往更直接、更像标准问答；Agent 因为加入证据、不确定性和复核步骤，有时答案更长，所以清晰度略低。

**2.4 Usefulness，实用性**

` Usefulness` 是实用性评分，范围是 `1-5` 分。

它衡量答案对实际工程分析是否有帮助，例如：

  * 是否给出可执行检查步骤
  * 是否说明该测哪个节点
  * 是否建议核对规格书、替换器件、测波形
  * 是否能帮助定位或修复问题
  * 是否避免泛泛而谈


如果答案只说“检查供电、检查接地、检查元件”，但没有指向具体节点或原因，`Usefulness` 会偏低。

**2.5 Factual Consistency，事实一致性**

` Factual Consistency` 是事实一致性评分，范围是 `0-1`。

它衡量答案有没有编造或扭曲事实，例如：

  * 是否编造了题目或图片中不存在的型号
  * 是否编造了测量值、波形、参数
  * 是否把“不确定/看不清”的内容说成确定事实
  * 是否引用了不存在的来源
  * 是否给出无证据支持的强因果结论


这个指标对 Agent 很重要，因为 Agent 的目标是基于证据回答，而不是自由发挥。你的结果里 Agent 的 `Factual Consistency` 高于 Baseline，说明证据流程有助于降低幻觉和错误扩展。

  3. **Semantic Similarity，语义相似度**


` Semantic Similarity` 衡量预测答案和参考答案在语义上的接近程度，范围通常是 `0-1`。

它不是简单看字面是否一样，而是看整体意思是否相近。当前系统优先使用 BERTScore，如果不可用，可以 fallback 到 sentence-transformer。

例如：

参考答案说“R9 和 C5 构成 RC 滤波，滤除 MOS 开通尖峰，避免误触发过流保护”。

预测答案说“该电阻电容网络用于抑制电流采样端的高频尖峰，使控制芯片不会误判过流”。

虽然字面不同，但语义接近，`Semantic Similarity` 会比较高。

  4. **ROUGE-L**


` ROUGE-L` 是基于最长公共子序列的文本重合指标，范围是 `0-1`。

它衡量预测答案和参考答案之间有多少较长的连续或相对有序表达是重合的。

它更偏字面匹配，例如：

  * 是否用了和参考答案类似的表达
  * 是否覆盖了相同的句子结构
  * 是否有相同的关键短语


优点是稳定、可复现；缺点是对同义改写不够友好。技术问答里，如果答案技术上正确但表达方式不同，`ROUGE-L` 可能仍然不高。

你的结果里 Baseline 的 `ROUGE-L` 高于 Agent，说明 Baseline 的答案和参考答案字面更接近。

  5. **Bigram Jaccard**


` Bigram Jaccard` 是双字/双词片段的 Jaccard 相似度，范围是 `0-1`。

简单理解，它会把文本拆成相邻的两个字符或两个词组成的小片段，然后比较预测答案和参考答案的重合比例。

它主要衡量：

  * 关键词是否一致
  * 元件名、型号、数字是否命中
  * 短语表达是否相似


例如参考答案里有“相位补偿”“高频滤波”“反馈环路”，预测答案也出现这些短语，Bigram Jaccard 会提高。

但它同样偏字面。如果答案说“环路稳定性补偿”而参考答案说“相位补偿”，语义接近但字面片段不同，分数可能不高。

  6. **Scoring Point Coverage，采分点覆盖率**


` Scoring Point Coverage` 衡量预测答案命中了多少参考答案中的关键采分点，范围是 `0-1`。

系统会从参考答案中抽取一些关键点，例如：

  * 关键元件：`R9`、`C5`、`MOS`、`NTC`
  * 关键数值：`311V`、`0.1uF`、`100Ω`
  * 关键机制：`RC滤波`、`相位补偿`、`浪涌限流`
  * 关键建议：`核对规格书`、`示波器测量`、`替换验证`


然后判断预测答案是否覆盖这些点。

例如参考答案有 8 个采分点，预测答案命中 5 个，则覆盖率约为：
```text
ScoringPointCoverage = 5 / 8 = 0.625
```

这个指标特别适合技术问答，因为很多答案不是要逐字一样，而是要把关键工程点说到。

## Agent 提升来源

Agent 相比 Baseline 的主要提升来源有三点。

第一，Agent 的证据流程提高了技术准确性。Agent 不是直接生成答案，而是通过图片检查、领域技能匹配、证据排序和最终合成来回答问题。共享样本上 Agent 的 Accuracy 从 3.2459 提升到 3.5656，Factual Consistency 从 0.6917 提升到 0.7423，说明这种证据驱动流程减少了关键结论偏差和事实不一致。

第二，Agent 对参考答案采分点覆盖更充分。共享样本上 Scoring Point Coverage 从 0.5313 提升到 0.5910，相对提升 11.25%。Completeness 从 2.7814 提升到 3.1749，相对提升 14.15%。这说明 Agent 更容易覆盖原因机制、关键器件、计算关系、检查步骤等评分所需信息。

第三，Agent 的语义质量更稳定。共享样本上 Semantic Similarity 从 0.7421 提升到 0.7665，逐样本胜率为 86.61%。这说明 Agent 生成的答案虽然不一定和参考答案字面一致，但整体技术含义更接近参考答案。

## Baseline 优势与原因分析

Baseline 在 ROUGE-L、Bigram Jaccard 和 Clarity 上优于 Agent：

  * ROUGE-L：Baseline 0.1696，Agent 0.1455。
  * Bigram Jaccard：Baseline 0.0890，Agent 0.0833。
  * Clarity：Baseline 4.6230，Agent 4.4153。


这说明 Baseline 的回答在表述上更直接、更清楚，也更容易复用题目或参考答案中的字面关键词。Agent 由于引入证据整合和不确定性说明，答案往往更长、更工程化，可能包含更多复核步骤、证据限定和补充解释，因此在字面重合指标和清晰度上略吃亏。
