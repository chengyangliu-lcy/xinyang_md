## 二、重点工作进展

### 2.1 联网搜索方案调研与技术选型

本月首先对 Agent 联网能力进行了系统调研。当前 Agent 需要具备两类核心联网能力：

  * **Web Search** ：根据用户问题检索候选网页；
  * **Web Fetch / Web Read** ：读取候选网页正文，并交给 LLM 进行分析、总结和引用。


原有方案主要依赖直接爬取 DuckDuckGo、Bing、Yahoo 等搜索引擎 HTML 页面，以及使用 `requests + BeautifulSoup` 读取网页正文。这种方式成本较低，但稳定性较差，主要问题包括：

  * 搜索引擎反爬导致请求失败；
  * 搜索页面结构变化导致解析不稳定；
  * JS 渲染页面无法通过普通 requests 获取完整正文；
  * 遇到 Cloudflare 校验页、403、429 限流时容易失败；
  * 只依赖搜索结果 snippet，无法获得完整网页证据。


针对这些问题，本月对多种免费或低成本方案进行了对比，包括 SearXNG、DuckDuckGo Python 库、Brave Search API、Tavily、Google Custom Search、SerpAPI、Exa AI 等。

最终选定 **SearXNG 自部署** 作为主要搜索入口，原因是：

  * 免费开源，无需商业 API Key；
  * 可通过 Docker 快速部署；
  * 支持聚合多个搜索引擎；
  * 提供 JSON API，便于返回结构化搜索结果；
  * 可以统一做结果清洗、去重和相关性过滤；
  * 相比直接爬取搜索引擎页面，稳定性和可维护性更好。

### 2.2 联网搜索功能实现、部署与测试

在完成方案调研后，本月已将 SearXNG 部署至公司服务器，并接入知识库问答平台。

当前系统在内部知识库无法充分回答问题时，可以调用外部联网搜索能力获取补充资料。搜索功能主要用于增强问答系统对实时信息、公开资料、技术资料、论坛内容和官方文档的覆盖能力。

当前搜索链路主要包括：
```text
用户问题
  ↓
知识库问答平台判断是否需要联网
  ↓
调用 SearXNG 搜索接口
  ↓
返回结构化搜索结果
  ↓
结果清洗、去重、过滤
  ↓
进入网页正文抓取流程
```

![image.png](https://cdn.nlark.com/yuque/0/2026/png/43223830/1780486174775-bde8003f-4b4a-4b19-9a5e-8d2de9f87423.png)

相比原先直接爬取 DuckDuckGo、Bing、Yahoo 等搜索页面，当前方案主要改进如下：

  * 减少对单一搜索引擎页面结构的依赖；
  * 降低反爬、超时、页面解析失败带来的影响；
  * 不依赖 Tavily、Brave、SerpAPI 等商业 API Key，免费；
  * 搜索结果以结构化形式返回，便于后续 Agent 处理；
  * 支持统一过滤低价值页面、站内搜索页、商品页、地图页等干扰结果。

### 2.3 Web Fetch 多后端网页正文抓取优化

本月整合了多种网页抓取后端，使系统能够对候选 URL 继续执行正文读取，并从多个抓取结果中选择质量最高的网页证据。

当前网页读取层采用多后端容错方案，主要包括：

  * **Jina Reader** ：适合快速获取网页 Markdown 内容；
  * **Crawl4AI** ：适合动态网页、技术文章和论坛内容提取；
  * **Scrapling** ：用于增强动态页面抓取能力，对部分论坛页效果更好；
  * **Turndown** ：通过 Node.js 将 HTML 转换为 Markdown，作为补充后端；
  * **requests + BeautifulSoup** ：作为轻量兜底方案；
  * **snippet fallback** ：对于 PDF、慢站点或抓取失败页面，保留搜索摘要，避免证据链为空。


优化后的链路如下：
```text
SearXNG 搜索候选 URL
  ↓
web_read 多后端抓取正文
  ↓
内容清洗
  ↓
质量评分
  ↓
选择最优网页证据
  ↓
交给 Agent / LLM 分析
```

该方案可以避免单一抓取器在论坛页面、动态网页、编码异常页面上失效，提高整体网页证据获取稳定性。

### 2.4 搜索与抓取链路测试验证

本月使用测试脚本对搜索与抓取链路进行了多组验证，覆盖场景包括：

  * 单样本搜索测试；
  * 多样本搜索 + fetch 测试；
  * 直接 URL 抓取测试；
  * 不同 fetch 后端对比测试；
  * 技术论坛、博客、datasheet 等不同类型页面抓取测试。


测试结果表明：

  * SearXNG 搜索链路可以替代原先不稳定的搜索引擎 HTML 爬取方式；
  * web fetch 能够从论坛、技术博客等页面中提取更完整正文；
  * competitive read 模式能够对多个抓取后端结果进行比较，并选择质量更高的正文；
  * 对 EEWorld 等论坛页面，直接抓取测试能够返回可用正文；
  * 对 datasheet、技术博客、电路问题分析等场景，新方案能够提供更完整、更可追溯的网页证据。

### 2.5 嘉立创/LCSC 元器件 MCP 工具设计与接入

本月围绕硬件研发场景，完成了嘉立创/LCSC 元器件 MCP 工具的设计方案，并已推进到 WebUI 平台接入和部署测试阶段。

#### 2.5.1 建设背景

在硬件开发、元器件选型和故障分析过程中，工程师经常需要查询元器件型号、规格参数、库存、价格、数据手册等信息。传统方式需要人工打开嘉立创或 LCSC 网站检索，效率较低。

因此，本月设计了一套面向 AI 助手的嘉立创元器件 MCP 工具，使芯阳 WebUI 平台、Claude Code、Codex、OpenCode 等 AI 助手能够直接接入嘉立创/LCSC 元器件数据源。

用户只需要通过自然语言提出查询需求，即可完成元器件搜索、详情查询和数据手册获取。

#### 2.5.2 数据源摸底

目前已完成嘉立创/LCSC 相关公开接口的初步摸底，确认以下接口可用于工具开发：

  1. **JLCPCB SMT 零件搜索接口**


  * 用途：按关键词搜索元器件；
  * 输入：keyword、pageSize、currentPage；
  * 输出：C 编号、名称、型号、厂商、分类、库存、阶梯价格、数据手册链接、Basic/Extended 标识、RoHS 标识等；
  * 当前状态：可用，无需认证。


  2. **LCSC 元器件详情接口**


  * 用途：根据 C 编号查询元器件详情；
  * 输入：productCode；
  * 输出：元器件名称、型号、厂商、分类、库存、最小包装数量、阶梯价格、数据手册直链、英文描述、完整技术参数列表等；
  * 当前状态：可用，无需认证。


  3. **数据手册获取能力**


  * 复用 LCSC 元器件详情接口；
  * 从详情返回结果中提取 pdfUrl 字段；
  * 封装为独立工具，方便用户直接获取官方 datasheet 链接。


#### 2.5.3 MCP 工具设计

计划实现 3 个 MCP 工具：

  1. **lcsc_search：元器件搜索**


  * 按关键词搜索嘉立创/LCSC 元器件库；
  * 返回 C 编号、型号、厂商、库存、价格、Basic/Extended 标识和数据手册链接；
  * 典型问题包括：


  * “帮我查一下 STM32F103C8T6”
  * “有没有 0402 100nF 电容”
  * “查一下 LM358 的可用型号和库存”


  2. **lcsc_detail：元器件详情查询**


  * 根据 LCSC C 编号查询单个元器件完整信息；
  * 返回型号、厂商、分类、库存、最小起订量、阶梯价格、数据手册和技术参数；
  * 支持用户只输入数字编号，工具自动补全 C 前缀。


  3. **lcsc_datasheet：数据手册获取**


  * 根据 LCSC C 编号获取元器件 PDF 数据手册链接；
  * 若元器件存在 datasheet，则返回官方 PDF 下载链接；
  * 若暂无数据手册，则返回明确提示。


#### 2.5.4 技术选型

当前工具采用以下技术方案：

方向| 选型| 原因  
---|---|---  
实现语言| TypeScript / Node.js| 生态成熟，便于 npm 安装和分发  
MCP SDK| @modelcontextprotocol/sdk| 官方 MCP SDK，适合实现标准 MCP Server  
传输方式| stdio| 便于被 WebUI、Claude Code、Codex、OpenCode 等工具调用  
部署方式| npm install -g| 安装后生成全局命令，便于复用  
工具形态| MCP Tool| 适合被 AI 助手通过自然语言自动调用
