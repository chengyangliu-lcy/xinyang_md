# 评测指标重设计进度汇报

## **详细实现说明**

下面这版可以直接写进汇报里，按“改了什么、怎么实现、为什么更合理、完全正确怎么判”来讲。

## 1\. 总体改动

本次把原来偏字面重合的评测，改成了更适合技术问答的三层结构：

  * `rouge_l` 改为 `claim_rouge_l`
  * `bigram_jaccard` 改为 `technical_entity_match`
  * `scoring_point_coverage` 改为结构化采分点覆盖


同时增加了几个更直接的质量诊断字段：

  * `fully_correct_rate`
  * `critical_error_rate`
  * `core_conclusion_hit_rate`
  * `unsupported_entity_rate`
  * `required_point_coverage`


这套设计的核心不是“让分数更高”，而是让分数更像“技术答案到底有没有真正答对”。

* * *

## 2\. `claim_rouge_l` 的技术实现

旧版 `rouge_l` 是对整篇答案做最长公共子序列匹配，主要看字面像不像。现在改成“按 claim 逐条匹配”。

### 实现流程

  * 先用标点、编号、列表符号把参考答案切成多个 claim。
  * 再用技术实体、数值、单位、否定、因果关系、长度等规则过滤低信息 claim。
  * 对每个 claim 单独计算它和预测答案之间的 token 级 ROUGE-L。
  * 最后对所有 claim 做加权平均，得到总分。


### 具体细节

  * `split_claims()` 会把参考答案拆成更小的语义块。
  * `_is_informative_claim()` 用来判断这个块值不值得算分。
  * `_claim_tokens()` 会把 claim 转成技术 token 加字符 bigram 的混合 token 集。
  * `_token_rouge_l()` 不是直接对原始字符串算，而是对 token 序列算 LCS，再结合 F-beta 和覆盖率。
  * `_claim_weight()` 会根据 claim 里的技术实体、关系词、否定词、数字和长度动态加权。
  * `claim_rouge_l.to_json()` 会额外输出 `claim_scores` 和 `claim_weights`，便于看每条 claim 怎么来的。


### 为什么更合理

  * 对同义改写更宽容。
  * 对答案顺序不敏感。
  * 不会因为答案多写了合理补充就被严重扣分。
  * 更关注“关键要点有没有覆盖”，不是“全文像不像参考答案”。


* * *

## 3\. `technical_entity_match` 的技术实现

旧版 `bigram_jaccard` 只是字符片段重合，没法真正理解技术对象。现在改成动态技术实体匹配。

### 实体抽取方式

当前实体不是靠一个固定小词表硬凑出来的，而是混合三类来源：

  * 强规则结构化实体
  * 英文/技术缩写
  * 动态中文技术短语


### 具体抽取对象

  * 元件编号：`R15`、`C323`、`U1`
  * 型号/料号：`TL431`、`LM358`
  * 数值和单位：`5V`、`10kΩ`、`100nF`
  * 技术缩写：`MOS`、`ESR`、`EMI`、`PWM`
  * 动态技术短语：通过 `jieba` 关键词、TextRank、CJK 片段兜底提取


### 匹配逻辑

  * 先从 `question_text + reference_answer + scoring point aliases` 构造 support 集。
  * 再抽取 reference 实体和 prediction 实体。
  * prediction 里的 keyphrase 不是全收，而是要么被 support 支持，要么权重足够高。
  * 最终分数不是简单 Jaccard，而是加权 `F_beta`，默认 `beta=1.5`，偏召回。
  * `unsupported_entities` 单独列出 prediction 里高风险但不被支持集覆盖的实体。


### 具体代码行为

  * `build_technical_idf()` 会基于全数据集的题面和参考答案建 IDF。
  * `_extract_entities()` 会按 `structured / domain_abbrev / keyphrase` 三类抽取。
  * `_limit_entities()` 会限制每类实体数量，避免长答案把实体数撑爆。
  * `_unsupported_entities()` 只保留高风险未支持实体，不把普通泛词混进去。
  * `technical_entity_match.to_json()` 会输出 `precision / recall / f1 / f_beta / unsupported_entity_rate` 等诊断字段。


### 为什么更合理

  * 能看出模型有没有提到正确元件、参数、型号。
  * 对中文表达变化更稳。
  * 能单独暴露“编造实体”。
  * 不会被普通字面重合误导。
  * 更适合电子故障诊断，而不是通用文本相似度。

## 4\. `scoring_point_coverage` 的技术实现

这是现在最关键的主评测之一。它不再是简单正则匹配，而是结构化采分点覆盖。

### 采分点结构

每个采分点都有这些字段：

  * `id`
  * `type`
  * `text`
  * `aliases`
  * `weight`
  * `required`
  * `evidence_source`
  * `point_type_confidence`


### 采分点类型

  * `core_conclusion`
  * `cause_mechanism`
  * `component_or_value`
  * `diagnostic_step`
  * `fix_suggestion`
  * `caveat`


### 采分点怎么抽

  * 先按句子和分号、标点切分参考答案。
  * 再对长句用 `claim_rouge_l` 的拆分逻辑进一步切成子 claim。
  * 用 `_information_score()` 评估这个片段是否足够“像一个采分点”。
  * `_classify_point()` 根据实体强度、因果关系、动作词、注意词等判断它属于哪一类。
  * `_build_structured_point()` 给每个点分配权重和是否 required。


### 命中怎么判

命中不是单一字面匹配，而是综合证据：

  * `exact`：整段精确命中
  * `entity_overlap`：技术实体重合度
  * `char_overlap`：字符 bigram 重合
  * `claim_overlap`：内容 token 覆盖率
  * `aliases`：参考点别名命中


状态判定阈值是：

  * `hit`：证据足够强
  * `partial`：有明显部分覆盖
  * `missed`：没覆盖
  * `contradicted`：出现明确矛盾


### 矛盾怎么判

当前不是用“只要有否定词就算矛盾”这种粗规则，而是更严格的证据组合：

  * 共享了高价值技术实体
  * 存在方向互斥，如上管/下管、开路/短路
  * 存在数值冲突
  * 存在明确极性冲突，如“无需/需要”“不能/能”“不是/是”


### 覆盖率怎么计算

  * 每个采分点按 `weight * status_score` 计分
  * `hit = 1.0`
  * `partial = 0.5`
  * `missed = 0`
  * `contradicted = -1`


最终：
```text
    coverage = clamp(sum(weight * status_score) / sum(weight), 0, 1)
```

另外还有：

  * `required_coverage`：只看 required 采分点
  * `core_conclusion_hit`：核心结论是否命中
  * `critical_errors`：核心结论矛盾等严重错误
  * `unsupported_key_tokens`：预测里高风险但未被支持的关键实体


### 为什么更合理

  * 能区分核心结论、原因机制、步骤和建议。
  * 支持部分命中，不是非黑即白。
  * 核心结论错了会单独扣出来。
  * 空采分点不再默认满分。
  * 每个样本都能看见命中证据和矛盾证据，便于复盘。

## 5\. `fully_correct` 的评判标准

这个指标不是单一分数，而是一个“完全正确”布尔判定。

### 判定逻辑

在 [evaluator/evaluate.py](D:/xinyang/xinyang/ComponentAnomalyAgent/evaluator/evaluate.py) 里，`_fully_correct()` 的逻辑是：

  * 不能有核心矛盾
  * 不能有 `critical_errors`
  * `coverage` 不能低于 `0.80`
  * `required_coverage` 不能低于 `0.80`
  * 如果核心结论没有命中，而且存在 required 采分点，也不能算完全正确


如果开启了 LLM Judge，还要额外满足：

  * `accuracy >= 4`
  * `completeness >= 4`
  * `factual_consistency >= 0.85`
  * `judge score >= 0.80`
  * 不能有 LLM Judge 识别出的 `critical_errors`


如果没开 LLM Judge，系统会走更保守的本地规则：

  * `coverage >= 0.90`


### 这个指标怎么理解

`fully_correct` 的目标非常严格，它代表的不是“答得还不错”，而是“关键技术事实基本都对，且没有核心错误”。

所以它低不一定说明模型一无是处，更多时候是说明：

  * 还有关键点没覆盖齐
  * 或者存在核心结论偏差
  * 或者本地模式下阈值本来就很保守


### `fully_correct_rate` 怎么来

就是整个评测集里 `fully_correct = true` 的比例。

## 6\. 这些改动带来的整体收益

这次改造后，评测从“看起来像不像参考答案”转成了“有没有覆盖关键技术判断”。

主要收益是：

  * 对合理改写更公平。
  * 对技术实体和数值更敏感。
  * 能区分核心正确、部分正确和关键错误。
  * 能单独看出编造实体和核心矛盾。
  * baseline 和 agent 是在同一套更公平的规则下比较。
