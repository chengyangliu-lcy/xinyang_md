# ComponentAnomalyAgent 项目说明

## 一、项目背景

`D:\xinyang\xinyang\ComponentAnomalyAgent` 是一个新项目，目标是构建一个面向**组件异常分析问答任务** 的 Agent 系统。系统输入为数据集：
```text
    D:\xinyang\xinyang\2025_dataset.jsonl
```

系统需要针对数据集中的样本，基于 Agent 的多步推理能力，通过任务理解、工具调用、联网搜索、信息整合与答案生成，输出高质量回答，并建立一套完整、可解释、可复现的自动评测体系。

本项目的目标不只是“让模型回答问题”，而是要构建一个具备以下能力的完整系统：

  1. 能读取并处理数据集；
  2. 能以 Agent 方式完成推理与工具调用；
  3. 能输出结构化答案与中间过程记录；
  4. 能对答案进行自动评测；
  5. 能与现有 baseline 对齐比较；
  6. 能进行模型评估、实验分析与结果复盘。


* * *

## 二、项目目标

本项目需要实现一个支持工具调用的智能 Agent，用于对 `2025_dataset.jsonl` 中的样本进行自动问答与异常分析。整体目标如下：

### 1\. 数据读取与样本解析

系统需要支持从 `2025_dataset.jsonl` 中逐条读取样本，并提取用于推理与评测的关键字段，包括但不限于：

  * 样本 ID
  * 问题内容
  * 输入上下文
  * 参考答案
  * 类别或其他元信息


系统应支持单条推理和批量推理两种模式。

### 2\. Agent 化问答

系统不能仅采用“单轮 LLM 直接回答”的方式，而应设计为一个可扩展的 Agent 流程。该流程至少包括：

  * 理解问题
  * 制定执行计划
  * 按需调用工具
  * 获取外部证据
  * 反思与补充检索
  * 整合信息并输出最终答案


### 3\. 工具调用与联网能力

Agent 应支持多种工具，以增强任务完成能力。至少需要考虑以下能力：

  * 本地文件读取
  * 数据集样本解析
  * 检索工具
  * Web 搜索
  * 网页内容抓取与摘要
  * 中间过程日志记录


系统应支持在必要时联网搜索公开信息，并将工具返回结果纳入推理上下文。

### 4\. 自动评测与模型比较

系统需要实现独立的评测模块，对预测答案与参考答案进行多维度打分，并支持：

  * 单样本评测
  * 批量评测
  * baseline 对比
  * 模型对比
  * 策略对比
  * 错误分析与报告生成


* * *

## 三、设计原则

项目整体设计建议参考现有 Agent 框架的工程思路，如 Claude Code、OpenHands、SWE-agent、OpenClaw 等，吸收其在任务规划、工具调用、上下文管理、可扩展性和评测闭环方面的优点。

本项目建议采用**模块化架构** ，避免将所有逻辑堆叠在单一脚本中。整体上应形成“主 Agent + 工具层 + 评测层”的清晰结构。

* * *

## 四、总体架构设计

### 1\. Agent Core

负责完成问答任务的主流程，建议包括以下核心组件：

  * `Planner`：根据输入问题生成执行计划；
  * `Executor`：执行计划并调用工具；
  * `Reflector`：判断当前证据是否充分，决定是否继续检索、纠错或重试；
  * `Answer Synthesizer`：整合工具结果和推理信息，生成最终答案；
  * `Memory / Context Manager`：维护上下文、历史操作记录和摘要压缩结果。


### 2\. Tools Layer

作为 Agent 的能力扩展层，建议设计为可插拔结构。初期至少支持：

  * `file_reader`：读取本地文件或样本内容；
  * `dataset_parser`：解析 `jsonl` 数据；
  * `retriever`：文本检索；
  * `web_search`：联网搜索；
  * `web_reader`：网页抓取与摘要；
  * `logger`：记录中间过程、工具调用轨迹和异常信息。


### 3\. Evaluation Layer

评测逻辑必须独立于主推理流程实现。该层负责：

  * 指标计算
  * baseline 对比
  * LLM Judge 评估
  * 采分点抽取与命中分析
  * 错题分析
  * 报告生成

## 五、输入与输出要求

### 输入

主输入数据集路径如下：
```text
    D:\xinyang\xinyang\2025_dataset.jsonl
```

每条样本应经过统一封装，形成标准输入格式，用于后续推理与评测。

### 输出

系统应输出两类结果。

#### 1\. 推理结果

每条样本至少输出以下内容：

#### 2\. 评测结果

  * 字符 Bigram Jaccard 分数
  * LLM Judge 分数
  * 采分点命中情况
  * 最终综合得分
  * 错误原因分析

## 六、评分体系设计

本项目最终评分采用加权综合方式，包含三项核心指标：

  * **语义相似度（60%）** ：核心指标，用于判断预测答案与参考答案在语义层面是否一致；
  * **ROUGE-L（25%）** ：辅助指标，用于衡量预测答案对参考答案主要表达片段的覆盖程度；
  * **字符 Bigram Jaccard（15%）** ：约束指标，用于保证关键词、数字、单位、实体名称等字面信息的准确性。


综合得分公式如下：
```text
    FinalScore = 0.60 * SemanticSimilarity
               + 0.25 * RougeL
               + 0.15 * BigramJaccard
```

* * *

## 七、语义相似度设计：优先考虑 BERTScore

在语义相似度实现上，建议重点考虑 **BERTScore** 的可行性，并优先作为语义相似度核心实现方案。

原因如下：

  1. 本项目更关注“回答的意思是否正确”，而不是表述是否完全一致；
  2. 在异常分析类任务中，正确答案可能存在大量同义表达、语序变化和概括性改写；
  3. 相比纯字符串重合指标，BERTScore 更适合评估“表达不同但语义接近”的答案。


因此，建议在评分模块中将语义相似度设计为可插拔实现，但第一阶段优先落地 BERTScore。例如：
```text
    SemanticSimilarity = BERTScore_F1
```

同时需要注意，BERTScore 不能替代所有约束：

  * 对数值、单位、比例、时间、型号等精确信息不够敏感；
  * 对关键词的字面正确性约束不足。


因此本项目不应只依赖 BERTScore，而应采用如下思路：

> 以 **BERTScore** 作为核心语义指标，以 **ROUGE-L** 和 **字符 Bigram Jaccard** 作为补充与约束指标。

* * *

## 八、字符 Bigram Jaccard 设计要求

字符 Bigram Jaccard 作为约束指标，主要用于保证答案中关键字面信息的准确性，尤其适用于以下内容：

  * 关键组件名称
  * 数值
  * 单位
  * 比例
  * 时间
  * 阈值
  * 状态词
  * 专有名词


该指标需要考虑中英文混合文本场景，建议实现时支持：

  * 文本标准化
  * 中英文统一预处理
  * Bigram 集合构建
  * 交并比计算
  * 对数字、单位、符号的保留策略


* * *

## 九、LLM Judge 评测设计

除基础指标外，建议增加独立的 LLM Judge 评测模块，用于辅助判断回答质量，并提供更强的可解释性。

LLM Judge 不建议替代主评分，而建议作为：

  * 辅助评分项
  * 分析项
  * 对比项
  * 错误解释项


Judge 可从以下维度进行评分：

  * 语义正确性
  * 关键信息完整性
  * 是否存在事实错误
  * 是否存在严重遗漏
  * 是否偏题
  * 表达是否清晰


Judge 输出建议包括：

  * 分数
  * 简要评语
  * 扣分原因
  * 是否命中关键采分点


* * *

## 十、采分点机制设计

为了让评测结果更合理、更可解释，项目需要实现“采分点”机制。

所谓采分点，是指从参考答案中提取的关键要素，用于判断预测答案是否覆盖了核心信息。采分点建议包括：

  * 核心结论
  * 异常原因
  * 关键组件名称
  * 关键指标或阈值
  * 数值与单位
  * 时间与状态
  * 建议动作或处理措施


系统应支持以下能力：

  1. 从参考答案中自动或半自动抽取采分点；
  2. 判断预测答案命中了哪些采分点；
  3. 输出命中与漏召情况；
  4. 计算采分点覆盖率；
  5. 将采分点结果用于辅助评测和错误分析。


每条样本最终建议输出：

  * 参考答案采分点列表
  * 预测答案命中的采分点
  * 未命中的采分点
  * 错误命中的干扰点
  * 采分点覆盖率

## 十一、baseline 设计要求

本项目测评必须以现有 `qwen_eval.py` 为 baseline 基线实现。也就是说，新系统的结果不应脱离当前已有评测体系，而应与 `qwen_eval.py` 在统一口径下进行比较。

具体要求如下：

  1. `qwen_eval.py` 作为当前默认 baseline；
  2. 新 Agent 系统的评测结果需要与 `qwen_eval.py` 使用相同数据集进行对照；
  3. 评测指标、输出格式和统计逻辑应尽量保持一致或可对齐；
  4. 后续任何优化版本都必须报告相对于 baseline 的提升或退化情况。


建议新增独立的 baseline 对比模块，用于输出：

  * baseline 总体得分
  * agent 总体得分
  * 各项指标对比
  * 提升样本
  * 退化样本
  * 错误分析报告


建议产出以下结果文件：

  * `baseline_score.json`
  * `agent_score.json`
  * `compare_report.json`
  * `error_analysis.json`
