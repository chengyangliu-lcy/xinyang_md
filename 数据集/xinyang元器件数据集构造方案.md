# xinyang元器件数据集构造方案

**1\. 先决定哪些元器件会被改**

  * 在正样本里，每个文件改 `2` 或 `3` 处：`eligible_indices >= 9` 时改 `3` 处，否则改 `2` 处
  * 候选元器件必须先通过筛选：


  * `comment` 能被 `parse_value()` 解析成“数值 + 单位”
  * `designator` 非空
  * `designator` 在当前文件里唯一


  * 真正从候选里挑哪几个，不是随机，而是按哈希排序后取前几个，保证**每次重跑都一致**


**2\. 先把所有候选值建索引**

  * 程序先把全数据集的可改元器件按两种方式分组：


  * `(参数族, 原始单位)` 分组，例如 `(capacitance, nF)`、`(resistance, K)`
  * `参数族` 分组，例如 `capacitance`、`resistance`


  * nF / uF / F → 电容族
  * uH / mH / H → 电感族
  * R / K / MΩ / ohm → 电阻族
  * mV / V / kV → 电压族
  * uA / mA / A → 电流族
  * Hz / kHz / MHz → 频率族
  * mW / W → 功率族
  * 这样后面生成错误值时，就能优先借“更像原值”的别的元器件数值


**3\. 错误值的优先级顺序** ​

它会按 4 层优先级找一个替代值：

  * **第 1 层：同文件、同参数族、同单位**


  * 例如原值是 `10uF`，优先找同一个文件里别的 `uF` 电容


  * **第 2 层：全数据集、同参数族、同单位**


  * 如果本文件里没有合适的，再去全数据集找别的 `uF` 电容


  * **第 3 层：同文件、同参数族、不同单位也可以**


  * 例如原值是 `100nF`，同文件里找 `10uF` 也可以
  * 但借来以后会**换算回原单位** 再写回，所以还是会写成 `nF`


  * **第 4 层：全数据集、同参数族**


  * 最后再从全局同类器件里找


**4\. 同一优先级里，候选怎么选**

  * 不是纯随机，也是按哈希排序，保证可复现：
  * 这样两个好处：


  * 同一份数据重跑不会变
  * 不需要额外保存随机种子


**5\. 借来的值怎么变成新的**` comment`  
假设原始元器件是：

  * 原值：`100nF`
  * 借来的候选值：`10uF`


程序不会直接把 `comment` 写成 `10uF`，而是：

  * 先取候选值的**归一化数值**
  * 再除以“原单位的倍率”
  * 然后用原单位重新渲染


所以：

  * `10uF = 1e-5 F`
  * 原单位 `nF` 的倍率是 `1e-9`
  * `1e-5 / 1e-9 = 10000`
  * 最终写回：`10000nF`


也就是说，**参数族保持不变，最终单位尽量保持原单位不变** 。

**6\. 什么样的候选会被过滤掉**  
即使找到了候选值，也不是直接就用，还会继续过滤：

  * 归一化后和原值一样，跳过


  * 避免等价值错误，例如 `100nF -> 0.1uF`


  * 渲染不出来，跳过


  * `render_comment()` 只接受比较简单的数字格式
  * 太长、科学计数法、过多小数位都会放弃


  * 新值字符串和原值完全一样，跳过


**7\. 如果 4 层都找不到合适的候选值**  
就进入兜底逻辑，用固定倍数扰动：

  * `10`
  * `0.1`
  * `2`
  * `0.5`
  * `4.7`
  * `0.47`
  * `22`
  * `0.22`


  


例如原值是 `10uF`：

  * 乘 `10` → `100uF`
  * 乘 `0.1` → `1uF`
  * 乘 `4.7` → `47uF`
  * 乘 `0.22` → `2.2uF`


这些倍数的目的，是尽量生成“像常见标称值”的错误，而不是胡乱改一个数字。

**8\. 为什么用这些倍数**  
它们基本对应电子元件常见的数量级和 E 系列风格：

  * `10 / 0.1`：量级错一位
  * `2 / 0.5`：翻倍/减半
  * `4.7 / 0.47 / 22 / 0.22`：常见阻容值风格


所以生成的错误会更像真实设计中的“标值抄错”，而不是随便乱写。

**9\. 生成错误后，样本里具体怎么写回** ​

  * 先拿到 `corrupted_comment`
  * 再写到对应元器件的 `comment`
  * 如果 `description` 里包含原始 `comment` 文本，就同步替换
  * 然后把注错前后的完整元器件对象都写进 `gold.errors`


**10\. 一个完整例子**  
假设某个元器件原来是：

  * `component = C12`
  * `comment = 10nF`


流程可能是：

  * 先在同文件找别的电容、同单位 `nF`
  * 找到某个候选 `120nF`
  * 比较归一化值：


  * `10nF = 1e-8 F`
  * `120nF = 1.2e-7 F`
  * 不相等，可以用


  * 按原单位写回，得到：


  * `corrupted_comment = 120nF`


`gold` 里就会记录：

  * `original_comment = 10nF`
  * `corrupted_comment = 120nF`
  * `normalized_original = 1e-8 F`
  * `normalized_corrupted = 1.2e-7 F`


  


  


**一句话总结**  
当前错误值生成逻辑不是“随机改数字”，而是：

  * 先找**同类元件的真实值** 来借
  * 尽量保持**参数族一致**
  * 尽量保持**原单位一致**
  * 禁止**等价值改写**
  * 实在借不到，再用**固定倍数** 做兜底扰动
