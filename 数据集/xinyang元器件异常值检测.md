# xinyang元器件异常值检测

是的，你这个判断基本对：**只给一份元器件列表，LLM 通常不知道完整设计意图** 。  
但这不等于它什么都做不了。更准确地说：

  * **它能做“字段一致性异常检测”** ：不需要知道电路功能，也能抓出很多错。
  * **它能做“弱语义推断”** ：大概猜这是哪类板子、哪些器件像电源/音频/传感器。
  * **它做不了“强功能正确性判断”** ：比如某个 10uF 到底该不该是 22uF，某个分压电阻是不是算错了——这类必须知道网络、芯片型号、所在功能块，最好还要 datasheet 或原理图。


你这份数据里，LLM 其实已经能大概猜到这是个什么方向的设计：  
有 `asrpro语音识别芯片`、`MIC1`、`8002A` 功放、`TPS61022` 升压、`甲醛传感器`，所以它能推测这是 **语音/音频 + 传感器 + 电源管理** 的板子。  
但它**不知道 C10 是 U3 的输出电容，还是某个音频去耦，还是传感器旁路** 。这就是为什么“只靠 BOM 判断数值对不对”会有上限。

* * *

## 先说结论：你这个任务不要做成“纯 LLM 判异常”

更稳的做法是：

**规则引擎做硬判定 + LLM 做解析和解释。**

也就是：

  1. 先把 `comment / description / designItemID / footprint` 里的值解析出来。
  2. 再做确定性的规则检查。
  3. LLM 只负责：


  * 抽取中文描述里的参数
  * 解释为什么可疑
  * 在信息不全时给“疑似异常”而不是硬判错


* * *

# 一、只看这份列表，已经能抓出的异常类型

## 1）硬异常：同一个器件内部自相矛盾

这类**不需要知道电路功能** ，就能直接判。

你这份里比较明显的有：

### C10

  * `designItemID`: `CGA0805X5R226M350MT`
  * `comment`: `10uF`
  * `description`: `容值:22uF`


这就是典型硬异常：**comment 和 description 不一致** 。

### C16

和 C10 一样，还是同类矛盾。

### C25

  * `designItemID`: `CGA0805X5R226M350MT`
  * `comment`: `1uF`
  * `description`: `容值:1uF`
  * 但这个 `designItemID` 跟 C21/C22/C23/C24/C26/C27 是同一个，而那些都是 `22uF`


所以 C25 极大概率错了。  
这类规则叫：

**同一 MPN 应映射到同一组关键属性**

* * *

###  C13

  * `designItemID`: `0.1u`
  * `comment`: `22u`


自相矛盾。

### C17

  * `designItemID`: `0.1u`
  * `comment`: `10u`


自相矛盾。

### C19

  * `designItemID`: `0.1u`
  * `comment`: `22u`


自相矛盾。

这几个一看就像**库维护/复制粘贴错误** 。

* * *

### L1

  * `designItemID`: `YSPI0520-1R0M`
  * `comment`: `1.5uH`
  * `description`: `电感值:1uH`


这也是明显冲突。  
先不管真实值是多少，至少**三个来源没对齐** 。

* * *

## 2）高概率异常：MPN 和 footprint 不匹配

这类通常也能抓，但最好标成“高概率”，别标 100% 硬错误，因为有时库命名习惯会有差异。

### 电容里很明显的：

  * `C7 / C8 / C11 / C12 / C14`
  * `designItemID` 里都带 `0603`
  * `footprint` 却是 `C0402`


这类非常可疑。

### 电阻里也有一串：

  * `R1 / R2 / R5 / R13 / R14 / R15 / R16 / R18 / R19 / R20`
  * `designItemID` 都像 `0603...`
  * `footprint` 却是 `R0402`


这也很可疑。

### R17

  * `designItemID`: `0603WAF4701T5E`
  * `comment`: 直接写成了 part number
  * `footprint`: `R0402`


这个至少有两个问题：

  1. **comment 没有人类可读值**
  2. ` 0603` vs `R0402` 可疑


* * *

## 3）低可信异常：信息不完整、无法校验

这类不是“错”，而是“不能信”。

例如：

  * `C9`: `CAP_0603`，description 空
  * `C18`: `designItemID=0.1u`，但没有正规料号
  * `COUT1 / COUT2`: 只有 `22uF`，没有真实 MPN
  * `R4 / R6 / R10 / R11 / R12`: 只有 `10k`
  * `R8 / R9`: `Res_0402` \+ `750K`
  * `Q2 / Q3 / Q5`: description 空
  * `U1 / U2 / U4`: 都比较像“说明文字/占位符”，不是规范料号


这些不一定错，但它们会让**后续自动校验、采购、替代料匹配、EDA 联动** 都变差。

* * *

# 二、只给 BOM，LLM 到底能做到哪一步

## 能做的

### 1\. 一致性检查

看这些字段是否互相打架：

  * `comment` vs `description`
  * `comment` vs `designItemID`
  * `designItemID` vs `footprint`
  * 同一 `designItemID` 是否对应多个不同值


这一步最稳，最适合现在就做。

* * *

### 2\. 群体模式异常

例如：

  * 同一个 MPN 在大多数地方都写 `22uF`，突然有一个地方写 `1uF`
  * 一组重复去耦电容中，只有一个写法不同
  * 一堆 0603 料里，只有几个被放成 0402 footprint


这类异常，哪怕不知道电路，也能抓出来。

* * *

### 3\. 粗粒度功能推断

比如你这份表里，模型能猜到存在：

  * 语音识别
  * 麦克风
  * 功放
  * 升压电源
  * 传感器


但这一步只能帮助排序可疑项，**不能证明元件值对不对** 。

* * *

## 做不到的

### 1\. 某个数值是否满足芯片应用电路

例如：

  * `TPS61022` 反馈分压是不是能得到目标输出电压
  * 某个 `22uF` 实际该不该用 `10uF`
  * 某个 MOS 的 `Vdss` 是否够
  * 某个 LED 串联电阻是不是限流合理


这些都需要至少知道：

  * 器件连接到哪些 net
  * 属于哪个芯片周边
  * 目标输入输出规格
  * datasheet 推荐值/公式


* * *

# 三、你真正应该怎么做：分三层检测

## 第 1 层：纯 BOM 层异常检测

这层不依赖原理图，直接用你现有数据就能做。

### 规则建议

#### 规则 A：同一器件多字段值冲突

比如电容值、额定电压、精度、封装。

#### 规则 B：同一 MPN 必须一致

同一个 `designItemID` 应该对应相同的：

  * value
  * tolerance
  * voltage/current rating
  * footprint


#### 规则 C：MPN 编码和 footprint 一致

例如 `0603` 料号不该大面积落到 `0402` footprint。

#### 规则 D：器件类型和值单位一致

  * `C` 应该是 F/nF/uF
  * `R` 应该是 Ω/kΩ/MΩ
  * `L` 应该是 H/uH
  * 二极管、MOS、IC 则检查关键额定值字段是否存在


#### 规则 E：占位符/非标准料号标记为低可信

像 `10k`、`0.1u`、`22uF`、`CAP_0603`、`Res_0402` 这种，不能当正式库料。

* * *

## 第 2 层：BOM 群体模式检测

这层用“同类器件的统计规律”抓异常。

### 方法

先把每个器件归一化成结构化特征，例如：
```text
    {
      "type": "C",
      "value": 22e-6,
      "tolerance": 0.2,
      "voltage": 35,
      "dielectric": "X5R",
      "package_code": "0805",
      "footprint": "C0805",
      "mpn": "CGA0805X5R226M350MT"
    }
```

然后按这些维度聚类：

  * 同一 `mpn`
  * 同一 `comment`
  * 同一 `footprint`
  * 同一类型器件群


再找“群体中的少数派”。

比如：

  * 同一 MPN 只有一个值不同
  * 同一电源去耦群里只有一个封装不同
  * 一串同类电阻只有一个没有 description


* * *

## 第 3 层：电路语义检测

这一步才是“数值是否真的正确”的核心。  
但你必须给模型更多上下文。

### 最少要补的数据

至少补这几个字段：

  * `nets`: 这个器件连到哪些网络
  * `ownerIC`: 这个器件服务哪个芯片
  * `role`: 它是什么角色  
比如 `input_bypass / output_bulk / feedback_upper / feedback_lower / led_series / gate_pullup`
  * `rail`: 所在电源轨  
比如 `VBAT / 3V3 / 5V_BOOST / GND`


例如把 C21 变成这样：
```text
    {
      "designator": "C21",
      "mpn": "CGA0805X5R226M350MT",
      "value": "22uF",
      "voltage": "35V",
      "footprint": "C0805",
      "ownerIC": "U3",
      "role": "output_bulk",
      "nets": ["VOUT", "GND"],
      "rail": "5V_BOOST"
    }
```

一旦有了这些信息，就能做强校验：

  * `TPS61022` 的反馈电阻能不能算出目标输出
  * 输入/输出电容是否满足推荐范围
  * MOS 的 `Vdss` 是否高于实际电压
  * LED 限流电阻是否合理
  * 去耦电容位置和值是否符合常见模式


* * *

# 四、最推荐的架构：不要让 LLM 直接判错

建议你这样设计系统：

## 模块 1：参数抽取器

把 `comment / description / designItemID` 统一抽成结构化字段。

这里 LLM 很有用，因为中文 description 格式不统一。  
但抽完以后就别再“自由推理”了。

* * *

## 模块 2：规则引擎

用硬规则判：

  * 值冲突
  * 封装冲突
  * 缺失字段
  * 同料号不一致
  * 非标准料号


这一步应该是主判定器。

* * *

## 模块 3：弱语义打分器

LLM 在这里做辅助：

  * 解释为什么可疑
  * 判断“像不像复制粘贴错误”
  * 在多个证据冲突时给出置信度


例如：
```text
    {
      "designator": "C25",
      "severity": "high",
      "reason": "同一 designItemID 在 C21/C22/C23/C24/C26/C27 中均对应 22uF，而 C25 被标成 1uF，疑似 comment/description 填写错误",
      "evidence": [
        "designItemID 相同",
        "群体多数值为 22uF",
        "该条记录与群体不一致"
      ]
    }
```

* * *

## 模块 4：电路语义校验器

拿到 netlist 或原理图后，再做真正的“值对不对”判断。

# 六、最关键的一句话

**只给 BOM，适合做“异常检测”；要判断“数值是否正确”，必须把任务升级成“器件 + 网络 + 角色 + datasheet 约束”的问题。**

所以你现在最好的路线不是：

让 LLM 直接看列表猜哪个值错了

而是：

先做结构化解析和一致性校验，再逐步补充 netlist / block role，让系统从“BOM 异常检测”升级到“电路语义校验”。

你这份 JSON 已经足够做第一版规则检测器了。下一步最值得做的是把它变成这种输入：
```text
    {
      "designator": "Rfb1",
      "type": "R",
      "value": "750k",
      "mpn": "xxx",
      "footprint": "R0402",
      "ownerIC": "U3",
      "role": "feedback_upper",
      "nets": ["VOUT", "FB"]
    }
```

这样模型才能开始真正判断“这个数值对不对”。
